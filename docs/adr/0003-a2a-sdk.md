# ADR-003 — Adopt `a2aproject/a2a-rs` as the A2A Wire-Types SDK

**Date:** 2026-05-30
**Status:** Accepted

---

## Context

The design spec (§2.1, §11.2, §11.3) requires pinning an official A2A Rust SDK that provides wire-compatible types for the A2A v1 protocol. The spec notes:

> "v1 pins A2A protocol v1 and adopts its current wire binding by depending on the official a2aproject/a2a-rs crate's generated ProtoJSON types rather than hand-writing wire types."

As of 2026-05-30, the following was verified against crates.io and the a2aproject/a2a-rs GitHub repository:

### Verified SDK facts (`a2a-lf` / `a2aproject/a2a-rs`)

| Property | Value |
|----------|-------|
| **crates.io package name** | `a2a-lf` (published from the `a2a` workspace crate in `a2aproject/a2a-rs`) |
| **Pinned version** | `=0.3.0` (latest as of 2026-05-30, published 2026-05-12) |
| **License** | Apache-2.0 |
| **Repository** | https://github.com/a2aproject/a2a-rs |
| **Description** | "A2A v1 protocol types and core definitions" |
| **Dependency alias** | `a2a = { package = "a2a-lf", version = "=0.3.0" }` |

The workspace also publishes `a2a-server-lf` (HTTP/SSE server built on axum) and `a2a-client-lf` (async client). We depend on `a2a-lf` (wire types only) and build our own HTTP/SSE server via axum, per the design spec's decision that the HTTP/SSE server is ours (spec §4.2, `InboundTransport` port). The `a2a-server-lf` crate is available if the Increment-3 evaluation concludes we should adopt the upstream server implementation.

### Exported `TaskState` enum (verified from source, `a2a/src/types.rs`)

The Rust enum variants and their ProtoJSON wire values are:

| Rust variant | Wire value (serde) |
|---|---|
| `TaskState::Unspecified` | `"TASK_STATE_UNSPECIFIED"` |
| `TaskState::Submitted` | `"TASK_STATE_SUBMITTED"` |
| `TaskState::Working` | `"TASK_STATE_WORKING"` |
| `TaskState::Completed` | `"TASK_STATE_COMPLETED"` |
| `TaskState::Failed` | `"TASK_STATE_FAILED"` |
| `TaskState::Canceled` | `"TASK_STATE_CANCELED"` |
| `TaskState::InputRequired` | `"TASK_STATE_INPUT_REQUIRED"` |
| `TaskState::Rejected` | `"TASK_STATE_REJECTED"` |
| `TaskState::AuthRequired` | `"TASK_STATE_AUTH_REQUIRED"` |

**Match with spec §2.1:** The spec assumed ProtoJSON uppercase enum names — these MATCH exactly. The Rust variant names differ (PascalCase: `TaskState::Submitted` not `TASK_STATE_SUBMITTED`) but the wire representation is identical.

Note: `a2a-lf` 0.3.0 includes `TaskState::Rejected` and `TaskState::Unspecified` which are not mentioned in spec §2.1. These are additional states in the A2A v1.0 specification that the spec did not enumerate; they do not contradict the spec's listed states.

### JSON-RPC method names (verified from source, `a2a/src/jsonrpc.rs`)

The actual JSON-RPC method name strings used on the wire are **PascalCase**, not the URL-path or colon-delimited style assumed in spec §2.1:

| Spec §2.1 assumption | Actual wire method name |
|---|---|
| `message:send` | `SendMessage` |
| `message:stream` | `SendStreamingMessage` |
| `tasks/{id}:get` | `GetTask` |
| `tasks/{id}:cancel` | `CancelTask` |
| `tasks/{id}:subscribe` | `SubscribeToTask` |

**This is a discrepancy.** The spec §2.1 uses URL-path + colon style that reflects HTTP REST thinking, but the A2A v1 specification's JSON-RPC binding uses PascalCase method names (per A2A specification §9.4, confirmed independently). The `a2a-lf` crate is correct; the spec §2.1's phrasing of the operation surface is informal shorthand, not literal wire method strings. Implementation must use the crate's `a2a::jsonrpc::methods::*` constants, not the spec's shorthand.

**Agent Card** is published at `/.well-known/agent.json` in the A2A v1 specification (not `/.well-known/agent-card.json` as spec §2.1 states — verify path at implementation time against the spec).

### ACP method names (verified from schema, `schema/schema.json`)

The actual JSON-RPC method strings confirmed to MATCH spec assumptions:

- `initialize` — handshake
- `session/new` — create session
- `session/prompt` — send prompt
- `session/update` — agent notification (streaming updates)
- `session/cancel` — cancellation notification
- `request_permission` — agent requests user authorization

Note: the method is `request_permission` (not `session/request_permission` as the spec phrases it). The spec §3.1 uses `session/request_permission` as informal shorthand; the wire method is `request_permission`.

---

## Decision

Adopt `a2a-lf =0.3.0` (from `a2aproject/a2a-rs`) as the A2A wire-types provider. Import as:

```toml
a2a = { package = "a2a-lf", version = "=0.3.0" }
```

Adopt `agent-client-protocol =0.12.1` (from `agentclientprotocol/rust-sdk`) as the ACP SDK.

Both versions are pinned with the `=` prefix in `[workspace.dependencies]` per the dependency-currency policy (design spec §11.2). `Cargo.lock` is committed.

The HTTP/SSE server for A2A inbound is built using `axum` directly (the `InboundTransport` port implementation in `bridge-a2a-inbound`). `a2a-server-lf` is not adopted in v1 but is available for the Increment-3 evaluation.

---

## Consequences

**Positive:**

- Wire types are generated from the official A2A proto schema; correct serialization is guaranteed without hand-maintenance.
- `TaskState`'s ProtoJSON wire values (`TASK_STATE_*`) match the spec exactly; no translation layer needed at the A2A boundary.
- Both licenses are Apache-2.0, fully compatible with the project license.
- Pinned versions guarantee reproducible builds; the `cargo deny` check enforces license constraints.

**Discrepancies flagged (no blocking impact):**

- Spec §2.1 uses informal operation shorthand (`message:send`, `tasks/{id}:get`) rather than literal wire method names. Implementation uses `a2a::jsonrpc::methods::SEND_MESSAGE`, `GET_TASK`, etc.
- Spec §3.1 says `session/request_permission`; wire method is `request_permission`. Use the crate's type for dispatch.
- `a2a-lf` includes `TaskState::Rejected` and `TaskState::Unspecified` that spec §2.1 didn't enumerate. Both are valid A2A v1 states; handle them in the error model (map `Rejected` → `BridgeError::PermissionDenied`, `Unspecified` → internal error).

**Upgrade cadence:**

Per spec §11.2, upgrade checks are scheduled monthly. Each SDK bump is its own PR. The golden-fixture suite (spec §10) is the regression gate for any binding change. Any protocol-version-level change (A2A v2, ACP v2) gets a new ADR.
