Operator control panel · design study

a2a-bridge — Operator UI

A cockpit for one operator supervising many agents. Every screen is a thin client over surfaces the bridge already exposes — permission suspend/resume, session/task control, /metrics, and the trace routes — so the UI decides nothing on its own. Realizes roadmap item H3-5.

recommended · ships now

Web?

The web UI's one real edge is charts — and /metrics is Prometheus already, so point Grafana at it and get history for free. A bespoke chart app would duplicate Grafana badly and force CORS / static-serving into the core. If a native page is ever wanted, it's a separate bridge-webui binary — charts only, no control plane.

02

The pager — a mobile companion

post-federation · build last

Mobile is the weakest form factor, justified by exactly one moment: approving a deferred permission while away from the desk before it times out and kills the run. So it's a pager with a yes/no button, not a third cockpit — scoped to approve/deny, glance, cost alarm, and emergency cancel. It's gated on roadmap H3-2 (federation: TLS, per-caller identity, pinned server fingerprint) and a push relay that carries a content-free doorbell while the control-plane RPC always returns over the authenticated tunnel.

03

Where each form factor earns its place

Primary · now

TUI

Lives where the operator lives. Thin ratatui client over existing seams; the whole control plane maps 1:1 onto CLI verbs. Ship the Inbox first — days of work, and it makes permission_policy="defer" livable.

Charts · free

Web → Grafana

Point Grafana at /metrics for historical cost & latency. A bespoke web app only earns its keep as a separate charts-only binary — never routes bolted onto serve.

Companion · later

Mobile

A pager for the away-from-desk approval, gated on federation (H3-2). Same bridge-client payloads as the TUI, a revocable per-device token, and generation stamping so a stale approve is rejected — never a misapplied grant.

Design principle throughout: read-mostly with an explicit, confirm-gated control plane. The UI contains zero orchestration — no retries, no workflow decisions, no auto-merge — and fails closed on any server-identity change. Full written spec, all six TUI screens, interaction model, and tech sketch live in a2a-bridge-operator-ui-design.md.
