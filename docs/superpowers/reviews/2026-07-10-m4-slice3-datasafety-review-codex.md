# M4 Slice 3 ‚Äî Data-Safety Review (codex gpt-5.5 xhigh, repo-verified)

**Findings**
- **WRONG ‚Äî Live detached task `turn_log` rows are misclassified as orphans.** Detached/batch `WorkflowRunContext` never sets `task_id`, default is `None`, and SQLite persists that nullable value. See `crates/bridge-workflow/src/executor.rs:25`, `crates/bridge-workflow/src/executor.rs:243`, `crates/bridge-coordinator/src/coordinator.rs:701`, `crates/bridge-coordinator/src/batch.rs:860`, `crates/bridge-store/src/sqlite.rs:790`. Scenario: traces on, default `artifact_retention_days=14`, a `Working` detached task has a completed node turn older than cutoff; orphan TTL deletes its `turn_log` row even though the task is crash-resumable. With `artifact_retention_max_bytes>0`, this can happen immediately.
- **WRONG ‚Äî Terminal-task purge does not clean actual detached-task turn rows.** Purging `DELETE FROM turn_log WHERE task_id IN victims` misses detached/batch rows with `task_id NULL`. Scenario: `artifact_retention_days=0`, `purge_terminal_tasks_days=7`, completed detached task is 8 days old; task row is deleted, but `/turns/:turn_id` still returns 200 because `turn_row()` only checks `turn_log`. See `crates/bridge-a2a-inbound/src/server.rs:1034`.
- **WRONG ‚Äî `completed_ms` is not a safe ‚Äúdone‚Äù marker.** `TurnFinished` upserts `completed_ms`; `UsageFinalized` is a later queued command, and `update_turn_usage` fails if the row is gone. See `crates/bridge-observ/src/lib.rs:271`, `crates/bridge-observ/src/lib.rs:288`, `crates/bridge-store/src/sqlite.rs:812`, `crates/bridge-store/src/sqlite.rs:842`. Scenario: `artifact_retention_max_bytes=1`; eviction runs after finished-row insert but before usage update, deleting cost/token data irreversibly.
- **SMELL ‚Äî The TaskRecord default-safety proof must be scoped to retention only.** Under the proposed defaults, retention should call only artifact TTL deletes against `task_journal`, `task_node_checkpoints`, and `turn_log`; size and terminal-task purge are skipped. But serve boot can still update `tasks` via resume under default config, independent of retention. See `bin/a2a-bridge/src/main.rs:6294` and `crates/bridge-coordinator/src/detached.rs:1594`.
- **SMELL ‚Äî The cited schema/status/cascade anchors are real, but incomplete for safety.** Status enum and `is_terminal()` match the explicit SQL list; `working_tasks()` scans only `working`; checkpoint/start/journal cascade, `turn_log` does not. See `crates/bridge-core/src/task_store.rs:14`, `crates/bridge-core/src/task_store.rs:44`, `crates/bridge-store/src/sqlite.rs:140`, `crates/bridge-store/src/sqlite.rs:150`, `crates/bridge-store/src/sqlite.rs:158`, `crates/bridge-store/src/sqlite.rs:165`.

**Other Checks**
- Counter rebuild from surviving `turn_log` rows is defensible as described, but only after the row-linkage and finalization races are fixed. See `bin/a2a-bridge/src/main.rs:6119`, `crates/bridge-observ/src/lib.rs:430`.
- Drill-down routes return 404 on missing task/artifact/turn rows as claimed, but the missed detached `turn_log` cleanup breaks the ‚Äúpurged task leaves 404‚Äù contract.

Verdict: **REDESIGN** ‚Äî fix task/turn linkage, add a durable turn-log finalization barrier, and rewrite retention predicates against the actual row ownership model.
