# MERGED DESIGN — podman support for the a2a-bridge

## Convergent spine (both architects, independently — high confidence)

1. **Zero required Rust changes for podman to function.** `SandboxConfig.runtime` is already the seam: one resolved value (`domain.rs` `runtime()`, default `"docker"`) flows through S3 allowlist validation (`registry.rs:96-106`), all the pure argv composers (`compose_sandbox`, `compose_container_rw`, `compose_verify`, `compose_sandbox_named`, sweep/reap/inspect helpers in `sandbox.rs`), the reaper (`reaper.rs`), the `:rw` backend, and the `containers` CLI (`main.rs:2449-2538`, which already iterates per-runtime and reaps idempotently across engines). Podman enters as data; no branch anywhere. This is consistent with the ADR-0013/0016 posture (containment is config-only) and the repo's own stated production target (`docs/containerized-agents.md:7-9` names rootless podman).
2. **Config = a separate example file**, `examples/a2a-bridge.containerized.podman.toml`: `runtime = "podman"` in **every** `[agents.sandbox]` block (including `impl`) **and** `[verify]`; `allowed_cmds = ["podman"]`; locked egress only. A distinct file also yields a distinct `container_owner = hash(config_path, mount, agent_id)` — cleanly separating docker-owned and podman-owned fleets on a dual-engine host, matching the established per-project-concurrency mechanism.
3. **Egress = a new shell script, not `podman compose`.** `compose.egress.yaml` stays untouched as the documented Docker path; a podman script reproduces the **same names** (`a2a-egress-internal`, `a2a-verify-egress` internal; `a2a-egress-external` routed; `a2a-egress-proxy`, `a2a-verify-proxy`) so the bridge config contract is unchanged. Both architects independently rejected compose-provider indirection under the security boundary.
4. **Same bring-up shape**, converged independently: idempotent `rm -f`/`--replace` → `create --name … --network <internal>` → `network connect <external>` → `start` (docker/podman `run` can't take two `--network`s; tinyproxy dials upstream lazily, so attach-then-start is safe). Networks created idempotently; proxy image + filter files shared by reference.
5. **The `[verify].runtime` allowlist gap is real and pre-existing** — both found, with citations, that `allowed_cmds` has exactly one enforcement consumer (`validate_sandbox` S3) and `VerifyConfig.runtime` (`config.rs:397`, parsed at `:378`/`:411`, consumed at `verify.rs:147-154`) is never checked. The podman increment makes it operationally dangerous: a half-converted config silently shells docker for verify, and on a dual-engine host can **false-pass** against Docker's image store. Both want it closed.
6. **Go-template dialect is the top unverifiable risk.** Two sites emit Docker's `{{.Label "key"}}`: `containers.rs:25` (`LIST_FORMAT` → operator CLI) and `sandbox.rs:222-235` (`managed_inspect_argv` → **boot-time lease recovery**, `reaper.rs:110-134`). If podman's dialect diverges, `parse_record`/`plan_recovery` skip malformed lines — fail-closed into *invisibility* (stale containers unrecovered/unreaped). Both gate it live rather than pretending the source proves compatibility; both pre-designed seam-local fallbacks (see Divergence D2).
7. **Validation = live gates, grounded in the checked filter files** (agent net: `tinyproxy.filter` hosts — `anthropic.com`, `chatgpt.com` for codex (not `api.openai.com`), kiro/cognito hosts; verify net: registries only, creds-XOR-registries preserved), plus spawn smoke, allowlist negative, reap/recovery, `:rw` implement + creds round-trip, `containers` CLI.
8. **Docs/runbook**: podman build commands for reader/toolchain/proxy, `podman machine` notes for macOS, the disjoint image/volume stores (→ Kiro device-flow re-mint is **required**, named volume `a2a-kiro-data` doesn't carry over), and the egress re-up after machine restart.
9. **Docker stays the default and unchanged**; podman is additive opt-in.

## Divergences — adjudicated

**D1. Verify-runtime gate: where and how.** Codex ships it as concrete mechanism in the main design; claude defers it to a hardening slice (upgraded to "no longer optional"). **Merge: codex's mechanism, claude's sequencing.** It is not required for podman to *function* (the field already parses and works), so it must not block slices 1–3 — but it ships in this increment as slice 4 because the increment is what makes the gap load-bearing. Mechanism (codex's, verified against the code):
- `VerifyConfig::runtime() -> &str` defaulting `"docker"`.
- `RegistryConfig::effective_allowed_cmds()` so the **default** allowlist (no `[registry]`, currently derived from agents only at `config.rs:734-753`) unions agent raw commands + sandbox runtimes + the verify runtime when `[verify]` exists — without this, the check would self-reject a registry-less podman-verify config (codex's catch).
- `VerifyToml::to_config_checked(&self, allowed_cmds)` used at **both** implement parse sites (`main.rs:~1286` and the `--resume` path `~1550`, which parse verify separately from `into_snapshot()` — codex verified this ownership split, which is why the check cannot live purely in `bridge-registry`).
- Severity: **reject into the existing `VerifyOutcome::ConfigError` path** (`main.rs:806`) — no container spawns; the security property is "a disallowed verify runtime never executes." Reject over claude's warn-option: a silent-wrong-engine verify is precisely the false-pass hazard, and ConfigError is an already-handled, non-crashing outcome.

**D2. Template-dialect contingency (built only if gates fail).** Claude: per-runtime template selection inside the two pure builders (smallest diff — they already take `runtime: &str`). Codex: replace the label readers with an inspect-based path (`ps -aq --filter label=…` → `inspect` IDs → parse labels from JSON; explicitly "no shell hacks"). **Merge: gate first (G4 covers recovery — the safety-relevant site; G6 the CLI). If only `.Label` fails, the per-runtime template fork is the minimal fix; if dialect problems surface anywhere else, jump to codex's inspect-JSON path, which eliminates the dialect dependency class entirely.** Either way the change is confined to `sandbox.rs`/`containers.rs`; `parse_record` stays or forks trivially; zero domain movement. Pre-designed, not pre-built.

**D3. Scope: which podman.** Claude (R5) sharply scopes this increment to **macOS `podman machine`** and rules Linux-rootless gates out (uid semantics differ; the B2b-1 lineage shows the macOS gate masks Linux breaks). Codex flags SELinux relabeling (`:z`/`:Z`) "only on Linux hosts" and floats a narrow structured mount-label option. **Merge: claude's scope statement wins — macOS machine in scope, Linux rootless explicitly a separate validation increment — and codex's SELinux note is recorded as that increment's first known concern.** No freeform runtime args now (both agree); a narrow structured label option only if Linux validation demands it. The design keeps the Linux increment cheap (same config, same script, re-run gates) — it just doesn't claim it.

**D4. Script shape.** Codex: single `deploy/containers/podman-egress.sh` with `up | status | down` (paths resolved from the script dir). Claude: `egress-up.sh`/`egress-down.sh` pair plus an explicit **post-condition contract**. **Merge: codex's single script (`status` is operationally useful; podman-specific naming is honest since Docker keeps compose) + claude's contract as its documented spec:** *three networks exist (two internal with no external route, one routed); both proxies run attached to (their internal net + external); each proxy reachable from its internal net at exactly the URL the bridge config states; the verify proxy serves the registries-only filter.* Compose and the script are two implementations of one contract; **G2 tests the contract**, doubling as the drift detector (~15-line drift surface).

**D5. Containerfile `FROM` qualification (claude-only, verified above).** Qualify only the registry bases: `reader.Containerfile:3` → `docker.io/library/node:24-slim`, `proxy.Containerfile:1` → `docker.io/library/debian:stable-slim` (Docker no-ops). **`toolchain.Containerfile:4` is `FROM a2a-agent-reader:latest` — a local reference; do NOT qualify it** (would point at a nonexistent registry image and break both engines). Instead: runbook documents the build order (reader → toolchain); if podman short-name resolution ever refuses it, the fix is the podman-only spelling `localhost/a2a-agent-reader:latest` — not to be applied while the file serves Docker too.

**D6. Internal-net DNS (claude-only, R8).** Podman's aardvark-dns historically didn't serve `--internal` networks (fixed netavark ≥1.6 / podman ≥4.5, version-sensitive). **G2's first probe is name-resolution of `a2a-egress-proxy` from the internal net**; fallback is pure config because `EgressPolicy::Locked{network, proxy, no_proxy}` carries the proxy as opaque data: create internal nets with `--subnet`, pin proxies with `--ip`, set `proxy = "http://<ip>:8888"` in the podman example. Name-based first (debuggability), IP second.

**D7. Verify containers are unmanaged (codex-only).** `compose_verify` calls `compose_sandbox` with no labels (`sandbox.rs:136`), so `containers list|reap` (which filters `a2a.managed=1`) never sees them. Existing behavior on Docker too — **the design must not promise verify cleanup via `containers reap`**, and the runbook says so.

**D8. `sync-creds.sh` (adjudicated above — both right).** Function is host-side and runtime-agnostic: no change (claude). But line 48's printed Kiro re-login hint hardcodes `docker run` (codex): make the message runtime-neutral or honor `CONTAINER_RUNTIME=podman`. Docs-slice nit.

**Complementary pieces folded in:** claude's two **sibling parse tests** pinning the new example (patterns at `main.rs:3341` and the R11-style full-`validate()` test at `:3548` — the latter live-exercises S3 with `allowed_cmds = ["podman"]` at zero runtime cost); claude's **`egress = "open"` slice-1 smoke** is legitimate first-class config (`domain.rs:105`, parse arm `config.rs:626`) — the shipped podman example stays locked-only; claude's **warn-level preflight** (`<runtime> info` at serve boot / `implement` start, names the runtime, suggests `podman machine start`) rides in slice 4; claude's note that the bridge resolves `podman` via PATH (launchd-launched serves need the PATH note); kiro+MCP stays host-only by existing validation (`config.rs:831-839`).

## Flow (unchanged — only the value changes)

Config parse (`SandboxToml.runtime`, `VerifyToml.runtime`) → S3 validate → spawn via `acp_program_argv` / `compose_container_rw` / `compose_verify` → reap/sweep/recovery/CLI with the captured per-entry runtime. Sweep targets are `(runtime, owner)` pairs and the end-guard iterates the distinct runtime set, so mixed docker+podman hosts stay correct. Egress infrastructure state stays outside the bridge entirely.

## Validation gates (macOS `podman machine`; Linux rootless out of scope)

- **G1 — build + spawn:** reader → toolchain → proxy build; **single-agent** workflow completes (Slice-A finding: `design`/`code-review` fan out to multiple agents). May use `egress = "open"` as a diagnostic config.
- **G2 — egress contract (the security gate):** from the internal net: proxy resolves+reachable by name (else pin IPs and flip the config); allowlisted host via proxy OK; non-allowlisted refused by tinyproxy; `curl --noproxy '*'` → no route. Repeat on `a2a-verify-egress` with the registries filter — hosts taken from the checked filter files, blocks include `github.com`/provider hosts respectively.
- **G3 — allowlist negative:** podman config + `allowed_cmds = ["docker"]` → `sandbox runtime not allowed: podman` (live S3, mirrored by the parse test). Slice-4 follow-on: verify-runtime mismatch → ConfigError, runner never called; and no-`[registry]` + verify-podman default-allows.
- **G4 — reap + recovery:** `podman events`-asserted container start (not echo); kill mid-turn → owner-scoped boot-sweep; end → 0 containers immediately; **includes a lease-recovery pass exercising `managed_inspect_argv`'s template under podman**.
- **G5 — `:rw` + uid + creds:** full `implement` e2e; host-side ownership sane (machine virtiofs uid translation lands Docker-Desktop-like; B2b-1's `safe.directory` round-trip absorbs it); a token refresh writes back through the writable creds bind.
- **G6 — `containers` CLI:** list/classify/reap podman-owned containers (pins `LIST_FORMAT` + the per-runtime loop).

## Risks (ranked)

① Template-dialect divergence — fails closed into recovery/CLI invisibility; gated twice (G4/G6); contingency seam-local (D2). ② Half-converted `[verify].runtime` — false-pass on dual-engine hosts; closed by slice 4. ③ Internal-net DNS — fails safe-but-down; config-only IP fallback. ④ Compose-vs-script drift — G2-as-contract catches. ⑤ Proxy loss across `podman machine` restart — fails closed; idempotent re-up documented (quadlet/systemd deferred to the Linux increment). Failure modes already survivable in code: missing binary/stopped machine → ENOENT → `AgentCrashed` + reap no-op; verify → failed result; `is_stale` biases false on error so a `logs --since` quirk degrades staleness detection without breaking reap correctness.

## Slices + build order

1. **Config + images + docs:** podman example + two pinning tests; the two registry-`FROM` qualifications (NOT toolchain); runbook section; `sync-creds.sh` message. *Exit: G3 + an open-egress spawn smoke (G1).*
2. **Egress:** `podman-egress.sh` implementing the post-condition contract; name-vs-IP decided by probe. *Exit: G2 in full, both nets.*
3. **Full loop:** kiro re-mint, creds round-trip, `implement` e2e, reap/recovery, CLI. *Exit: G4+G5+G6 → **podman support ships here.***
4. **Hardening (recommended, this increment):** verify-runtime gate (codex mechanism, reject→ConfigError) + `effective_allowed_cmds()` + preflight probe; the template fork **only if** G4/G6 demanded it.

Effort ~2–3 days, dominated by validation. A third engine (`nerdctl`) would land as slices 1–2 of config work — the seam held under both architects' independent inspection.

## DECISIONS FOR THE OWNER

1. **Verify-runtime gate severity:** reject-into-`ConfigError` (codex; verify never runs on a disallowed runtime) vs warn-and-proceed (claude's softer option). **Recommend reject** — the silent-wrong-engine false-pass is the hazard, and ConfigError is an already-handled outcome; note it interacts with the B2b-3b tweak loop (a ConfigError verify can't contribute PASS, so a misconfigured loop will surface loudly rather than converge falsely).
2. **Linux-rootless deferral:** docs name rootless-podman-on-Linux as the production target, but this increment validates only macOS `podman machine` and the two differ on uid/SELinux semantics. **Recommend confirming the deferral** to a follow-up increment (carrying the SELinux `:z/:Z` structured-option question with it) rather than stretching this one.
3. **Template contingency pre-commitment (only if G4/G6 fail):** per-runtime template fork (smallest diff) vs inspect-JSON label reading (eliminates the dialect class). **Recommend:** template fork if `.Label` is the sole failure; inspect-JSON if anything else in the template surface diverges.
