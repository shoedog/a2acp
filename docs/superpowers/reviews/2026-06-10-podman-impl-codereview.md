Merged review — podman support (slices 1, 2, 4). Both lenses reported successfully; findings de-duplicated below. Severity disagreements resolved inline.

**BLOCKER 1 — preflight executes unvalidated runtimes** (`bin/a2a-bridge/src/main.rs:859`; call sites `1339`, `1611`, `1770`, `2787`). `preflight_runtimes` runs `<runtime> info` *before* `Registry::new` enforces `allowed_cmds`: with `runtime="/tmp/evil"` and `allowed_cmds=["podman"]`, the probe executes `/tmp/evil info` (main.rs:847/866) and only later would registry validation reject it (registry.rs:102). Config-driven arbitrary-binary execution defeats the allowlist's purpose. **Fix:** validate the snapshot before preflight, or have preflight probe only allowlisted runtimes. (Codex-unique; Claude's "no blockers" verdict missed this — Codex is right, the allowlist is the project's hard gate and preflight runs outside it.)

**BLOCKER 2 — `ensure_network` never verifies `--internal`** (`deploy/containers/podman-egress.sh:35-44`). `network exists` accepts any pre-existing network by name; if `a2a-egress-internal` was created without `--internal` (manual run, older tooling, collision), `up` prints success while agents have direct egress — the containment invariant the whole design rests on is silently void. G2 is a sometimes-run live gate; the script asserts the topology on every `up`. **Fix:** `network inspect` and assert `internal: true` on both internal nets; fail loud or recreate on mismatch. (Codex rated BLOCKER, Claude MAJOR — Codex is right: this is a silent security-boundary bypass, not a hygiene issue.)

**MAJOR 3 — verify-runtime gate is call-site convention, not construction-enforced** (`config.rs:443`, wired at `main.rs:1338`/`1610`). The gated and ungated `Option<Result<VerifyConfig>>` are the same type, so a future implement-like entry point (e.g. a serve-driven detached implement) that re-derives `verify_cfg` silently runs verify on a disallowed engine. **Fix:** gate inside `into_snapshot`, or check `allowed_cmds` at the `run_verify_step` consumption choke point. (Claude-unique.)

**MAJOR 4 — the `"docker"` default is triplicated** (`domain.rs:86`, `config.rs:451` "pin" comment, `config.rs:772` default-union). Nothing ties the gate's default to the one `compose_sandbox` applies later; flipping the project default desynchronizes them → spurious reject or, inverse, a disallowed engine passing the gate. **Fix:** a single `pub const DEFAULT_RUNTIME` in `bridge_core::domain` — or better, have the gate normalize (`vc.runtime = Some(resolved)`) so downstream never re-defaults; this also fixes MINOR 9 for free. (Claude-unique.)

**MAJOR 5 — `runtime_responds` is unbounded and blocking inside async** (`main.rs:846-853`; placement at `1339`/`1611`/`1770`/`2787`). A wedged `podman info` (half-started machine) turns the warn-only preflight into a startup hang for `serve`, `run-workflow`, and `implement`; it's also a blocking `std::process::Command` inside async fns, and serve's runtime-mutable registry means post-boot agents are never preflighted. **Fix:** bound the probe with a short timeout (warn on timeout), use `tokio::process`/`spawn_blocking`, and hang preflight off snapshot application so all paths and post-boot mutations share one seam. (Codex's hang + Claude's placement, merged — same fix.)

**MAJOR 6 — parity test misses structural drift and never runs full registry validation** (`main.rs:3366-3411`). The trimmed line-set diff plus the `docker`/`podman` token allowance demonstrably passes real breaks: changing `claude` to `access = "rw"` passes (both strings exist elsewhere) yet boot fails at `registry.rs:150-158`; duplicated lines (`proxy =`, `mount =`) make delete-one/move drift invisible; an `image = "docker.io/..."` swap sails through the token allowance; and the snapshot never goes through `Registry::new`. **Fix:** run full registry validation and compare the two *parsed* snapshots with only runtime/allowlist fields masked. (Codex MAJOR over Claude MINOR — the rw counterexample shows the test fails its stated purpose.)

**MAJOR 7 — build-if-absent leaves the agent proxy's baked filter permanently stale** (`podman-egress.sh:46-47`). The provider allowlist is baked via `COPY tinyproxy.filter`, but the script builds only when the image is absent — an operator who tightens the filter and re-runs `up` still serves the old allowlist; only the bind-mounted verify filter stays fresh. **Fix:** always `build` on `up` (layer-cached, cheap) or bind-mount the agent filter like the verify one. (Claude-unique.)

**MAJOR 8 — the IP-pinning fallback is un-pinned by its own recovery path** (`podman-egress.sh:22-29` header vs the `up` body). An operator on old podman pins subnets/IPs by hand and sets `proxy = "http://10.89.0.2:8888"`; the mandated re-`up` after every machine restart recreates proxies *without* `--subnet`/`--ip`, so the pinned config points at an arbitrary allocation — egress breaks exactly for the configs the fallback exists to serve. **Fix:** make pinning a script mode (e.g. `PIN_IPS=1`) so re-up reproduces the pinned topology. (Claude-unique.)

**MINOR 9 — defaulted verify runtime is never preflighted** (`main.rs:1341-1345`, `1613-1617`). The gate resolves `None → "docker"` internally and discards it; preflight re-extracts the raw `Option`, so a defaulted-docker verify under host-only agents probes nothing and docker-missing surfaces only as a confusing spawn failure at verify time. **Fix:** pass `v.runtime.as_deref().unwrap_or("docker")` — or get it free from the normalize-in-gate fix (MAJOR 4). (Both reviewers, deduped.)

**MINOR 10 — `down` masks in-use failures** (`podman-egress.sh:73-77`). `network rm ... || true` swallows "network in use" (the unmanaged verify containers the docs say exist) and prints "egress down" with networks still up. **Fix:** tolerate only not-found; report any other failure. (Claude-unique.)

**MINOR 11 — `up` is idempotent but disruptive, plus a stale comment** (`podman-egress.sh:49/55`; podman example `[registry]` block). Every `up` `rm -f`'s healthy running proxies — re-running mid-workflow blacks out agent egress for the rebuild window. Separately, the podman example's `[registry]` comment still reads "The spawned program is `docker`" — a drift class the parity test's comment-allowance guarantees persists (fixed by MAJOR 6's structural comparison). **Fix:** short-circuit when proxies are running and correctly attached (or document the disruption), and correct the comment. (Claude-unique.)

**Verdict: REJECT — ship after fixing the 2 BLOCKERs (preflight-before-validation, unverified `--internal`); MAJORs 3-6 should land in the same slice since they guard the gate's and test's stated invariants.**
