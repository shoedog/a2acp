**BLOCKER — podman-egress.sh:6, :59-61, :86-89: the `up` short-circuit trusts "running" as "correctly wired," silently voiding the egress contract.** (Merged: codex's blocker + claude's New-1/New-2 — same root cause, two facets.) (a) Any running container named `a2a-egress-proxy`/`a2a-verify-proxy` — left by an older script revision or other tooling, possibly missing the external-net attach or the filter bind-mount — passes `proxy_running` and `up` reports success while serving stale or unfiltered egress; the comment's "always attaches both nets" claim only holds for containers this script revision created. (b) The header promises "edit the filter, re-run `up`, no rebuild," but tinyproxy loads filter rules at startup — the bind mount refreshes the file, not the in-memory rules — so an operator who tightens the allowlist gets "egress already up" while the old permissive rules stay active until FORCE=1 or a machine restart. Fix: before short-circuiting, verify wiring (inspect image, network attachments, filter Mounts) and filter freshness (filter mtime vs container `StartedAt`), recreating on any mismatch; at minimum, change the header and the short-circuit message to mandate `FORCE=1 … up` after a filter edit. Codex's BLOCKER severity is right over claude's MAJOR/MINOR split: this script's only job is the containment boundary, and `up` can report a contained state that isn't — silent security regression beats doc inconsistency.
