# A2A Bridge Increment 3a (Conformant ACP Client via the SDK) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace the non-conformant hand-rolled ACP driver with a real, conformant, bidirectional ACP client built on `agent-client-protocol` =0.12.1 (`AcpBackend`), validated against real `kiro-cli acp` and `codex-acp`.

**Architecture:** `AcpBackend` (bridge-acp) wraps a `Supervised` child in the SDK's client-side connection (a bidirectional JSON-RPC peer). It drives the conformant lifecycle (initialize → lazy agent-minted session/new → prompt:[ContentBlock] → agent_message_chunk stream → result), owns a serialized bridge↔agent sessionId map with a cancel-latch, and services reverse requests (request_permission via PolicyEngine; fs/terminal unsupported). Conformance is CI-proven by wire-level golden assertions + a captured real-agent frame corpus.

**Tech Stack:** Rust 2021 (1.94.0), `tokio`, `agent-client-protocol` =0.12.1, `serde_json`, `futures`.

**Spec:** `docs/superpowers/specs/2026-05-30-a2a-bridge-v3a-acp-sdk-design.md` (incl. §11 research appendix). Tags `[Cx]`/`[Cl]` trace to accepted review findings.

**Branch:** create `feat/v3a-acp-sdk` off `main` before Task 1.

## CROSS-CUTTING RULES (apply to every task)
- **Keep the workspace green every task** `[Cx-M1/Cl-M2]`. **Transport-generic core (T0 decision):** the connection logic is generic over the SDK transport (it drives `Client::builder()…connect_with(transport, |cx| …)` — see "Confirmed SDK API" below — over *any* transport). `AcpBackend` keeps BOTH constructors throughout: **`spawn(cmd,args,AcpConfig)`** = production, where our `Supervised` owns the child (own process group, tested `SIGTERM→SIGKILL` reap, zero-zombie — matters because `codex-acp` forks subprocesses) and we feed the child's stdin/stdout to the SDK via a `ByteStreams`/`Lines` transport (likely a `tokio_util::compat` `.compat()` shim from tokio pipes to the SDK's futures-io); and **`connect(transport,AcpConfig)`** = accepts any SDK transport, used by the in-process fake-agent unit tests (`Channel::duplex()`, no real process) and trivially supporting an `AcpAgent::from_str`-spawned child later. (`spawn` is implemented as: build `Supervised`, then `connect(ByteStreams over its pipes)`.) The SDK rewrite replaces the *innards* incrementally; **never leave `prompt`/`cancel` as `todo!()` across a task boundary** — each task ends with `cargo test --workspace` green. Inline scripted-`/bin/sh` tests in `acp_backend.rs` are replaced **in the same task that rewrites the code they cover** (Tasks 2–4), not deferred to T8.
- **No tautology in golden tests** `[Cx-M3/Cl-M3/nit]`. Wire-golden tests capture the **complete outbound JSON-RPC line that the SDK actually writes to the child's stdin** (tap the transport / a pipe), and compare to **expected JSON authored by hand from §11A / the ACP protocol** — NEVER author the expected value by pasting `serde_json::to_value(<sdk type>)`. That would assert the SDK against itself.
- **Conformance Definition-of-Done (merge gate)** `[Cx-M3/Cl-M3]`: 3a is NOT done until **at least one REAL frame captured from `kiro-cli acp` AND one from `codex-acp`** is in the corpus (T7) and replays correctly, OR a passing T9 gated e2e run against each real agent is recorded. Hand-authored §11A frames are *provisional scaffolding*, explicitly marked, and do NOT by themselves satisfy the gate (golden + corpus sharing §11A as the only source of truth is the v1 failure mode).
- **Per-session prompt-turn serialization** `[Cx-M2]`: an ACP session handles one prompt turn at a time (`session/update` is keyed by session_id, not request id). `AcpBackend` holds a per-session turn lock; a second `prompt` for a session whose turn is in flight **waits** (serialized) — concurrent prompts must NOT be demuxed into two streams on one agent session.
- **Confirmed SDK API (T0 — done, resolved `agent-client-protocol v0.12.1` / schema `v0.13.2`).** There is **NO `ClientSideConnection` and NO handler trait** — the plan's old assumption is wrong. Build against this:
  - **Connection:** `Client.builder().name("a2a-bridge").on_receive_notification(handler, on_receive_notification!()).on_receive_request(handler, on_receive_request!()).connect_with(transport, |cx: ConnectionTo<Agent>| async move { /* drive lifecycle here */ }).await`. `connect_with` **owns the single-task event loop** (does not return until the connection closes — so spawn it as a long-lived task per backend); `cx: ConnectionTo<Agent>` is the cloneable handle for agent-bound calls. Transport for prod = `ByteStreams::new(writer, reader)` / `Lines::new(sink, stream)` over the `Supervised` child's pipes; for tests = `Channel::duplex()`.
  - **Handlers are closures, not trait methods.** `session/update` → `.on_receive_notification(async |n: SessionNotification, cx| {…}, on_receive_notification!())`; `session/request_permission` → `.on_receive_request(async |req: RequestPermissionRequest, responder: Responder<RequestPermissionResponse>, cx| { responder.respond(…); Ok(()) }, on_receive_request!())`. **CRITICAL:** call `responder.respond(...)` **synchronously** — calling `.block_task()` inside a handler **deadlocks** the event loop (this is the T5 mid-prompt risk; offload via `cx.spawn(...)` if a handler needs to await an agent call). Unregistered `fs/*`/`terminal/*` requests auto-return method-not-found — advertise no caps (`ClientCapabilities::default()`) and don't register them.
  - **Agent-bound calls:** `cx.send_request(<Req>).block_task().await?` (from the `connect_with` body or a spawned task, never a handler). `initialize` = `InitializeRequest::new(ProtocolVersion::V1)`→`InitializeResponse`; `session/new` = `NewSessionRequest::new(cwd).mcp_servers(vec![])`→`NewSessionResponse{session_id, modes, …}`; `session/prompt` = `PromptRequest::new(session_id, vec![ContentBlock::Text(TextContent::new(t))])`→`PromptResponse{stop_reason: StopReason}`; `session/set_mode` = `SetSessionModeRequest::new(session_id, mode_id)`; `session/set_model` = `SetSessionModelRequest::new(…)` **(needs feature `unstable_session_model`)**; `authenticate` = `AuthenticateRequest::new(method_id)` (basic `AuthMethod::Agent` stable; `EnvVar`/`Terminal` need `unstable_auth_methods`). **`session/cancel` is a NOTIFICATION:** `cx.send_notification(CancelNotification::new(session_id))` — not a request.
  - **Streaming:** `SessionUpdate::AgentMessageChunk(ContentChunk)` arrives via the notification handler while the `PromptRequest` future is still pending → route chunks to an `mpsc` the `BackendStream` reads; `StopReason` ∈ `EndTurn|MaxTokens|MaxTurnRequests|Refusal|Cancelled`. (High-level `ActiveSession`/`read_update()` wraps this mpsc — implementer may use it if it fits the `AgentBackend` streaming shape.)
  - **Permission types:** `RequestPermissionRequest{session_id, tool_call, options: Vec<PermissionOption>}`; `PermissionOption{option_id, name, kind}`; `PermissionOptionKind ∈ AllowOnce|AllowAlways|RejectOnce|RejectAlways` (PascalCase enum; wire is snake_case); reply `RequestPermissionResponse::new(RequestPermissionOutcome::Selected(SelectedPermissionOutcome::new(option_id)))` or `::Cancelled`.
  - **Wire-golden capture:** `AcpAgent::with_debug(|line, LineDirection::Stdin|…)` tees outbound JSON-RPC lines; for `connect`/duplex tests, tap the channel/`Lines` sink.
  - **Cargo features:** default covers all driven methods EXCEPT `set_model`. Set the dep to `agent-client-protocol = { workspace = true, features = ["unstable_session_model"] }` (T6 needs it). `authenticate` basic is stable; add `unstable_auth_methods` only if a real agent requires `EnvVar`/`Terminal` auth (defer until T6/T9 shows it).
