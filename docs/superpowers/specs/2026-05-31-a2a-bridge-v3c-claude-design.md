**Tech stack:** Rust 2021 (1.94), `tokio`, `serde_json`, `Supervised` (3a), `bridge-core`/`bridge-registry`. `claude` CLI 2.1.159 (installed, logged in). No HTTP deps (B1 deferred).

## 1. Why this shape (and an honest conductor framing)

ADR-0005 deferred the conductor fork/continue decision to "post-3c, second protocol family." 3c delivers a **second `AgentBackend` impl** (non-ACP) and a **second adapter kind** (`claude-cli`), but it is still a **stdio child process** — so it does NOT exercise the *non-process* dimension by itself. The `kind`-seam designs in a `claude-api` (B1) arm to carry that dimension. **Honest caveat (review):** `AgentEntry` currently *requires* `cmd` and `allowed_cmds` validation is exec-launch vocabulary — so a non-process `claude-api` (no `cmd`/`args`/`cwd`) **will** force making `cmd` kind-conditional/optional and reworking the `allowed_cmds` gate. That forced change is NOT "no domain change"; **that forced change IS the conductor signal** the post-3c re-eval weighs. The designed-in `claude-api` dispatch arm is commented (reasoned, not compiled) — weaker evidence than a built one. So 3c's conductor contribution is: (a) prove the ports absorb a non-ACP **process** backend cleanly (claude-cli), and (b) surface the exec-centric `cmd`/`allowed_cmds` residue a non-process backend must change — *named explicitly* for the re-eval.

**Chosen path (B2):** warm Claude Code — reuses the subscription login (no API key, unlike A and B1), genuine interactive multi-turn, fits the warm-backend model. Rejected/deferred: **A** (ACP-Claude adapter — same family AND needs an API key, rejecting subscription OAuth) and **B1** (Anthropic API direct — strongest non-process test but needs an API key) — both designed-in via the kind-seam for later.

## 2. The adapter-`kind` seam

`bridge_core::domain::AgentEntry` gains `kind: AgentKind`:
```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq, Default)]
pub enum AgentKind { #[default] Acp, ClaudeCli /* , ClaudeApi (future B1) */ }
```
TOML `[[agents]]` gains `kind = "acp" | "claude-cli"`, **defaulting to `acp`** (back-compat). **Parse as `Option<String>` on `AgentEntryToml` and map in `into_snapshot`** (like `effort`, `config.rs`) so an unknown value yields `ConfigError::Registry` (not a raw TOML deserialize error) `[review]`.

**Backend factory dispatches on kind** (3b's `SpawnFn` in `bin/a2a-bridge/src/main.rs` becomes a dispatcher):
```rust
match entry.kind {
    AgentKind::Acp       => AcpBackend::spawn(&entry.cmd, &args, acp_config(&entry)?).await?.with_policy(policy),
    AgentKind::ClaudeCli => ClaudeCliBackend::spawn(&entry, policy).await?,   // warm pool inside
    // AgentKind::ClaudeApi => …                                            // B1: NO Supervised child (forces cmd-optional — §1)
}
```
The `bridge-registry` crate stays kind-agnostic (it calls the `SpawnFn`, holds `Arc<dyn AgentBackend>`; lease/binding/retire machinery is transport-agnostic). **`kind` MUST join the slot-reuse identity** `[review]`: 3b's `apply()` keeps a slot warm when `cmd/args/cwd/auth_method` are unchanged — add `kind` to that tuple, so an `acp`↔`claude-cli` edit (both `cmd="claude"`) forces a fresh cold slot, not a same-backend config-edit. `allowed_cmds` still gates `cmd` (`claude-cli` uses `cmd="claude"` → add `"claude"`).

```toml
[registry]
allowed_cmds = ["kiro-cli", "codex-acp", "claude"]
[[agents]]
id = "claude"
kind = "claude-cli"
cmd  = "claude"
args = []                       # extra flags (perm/trust); §4
model = "claude-opus-4-8"       # → --model at warm spawn (best-effort)
cwd  = "/abs/trusted/dir"       # absolute; trust gate §4
model_provider = "anthropic"    # descriptive
[agents.extensions]
  idle_ttl_secs = 3300          # idle reap, ≈ Max prompt-cache window (default 3300 = 55 min; §3.3)
  max_warm = 16                 # idle-retention cap — soft, idle procs only (default 16; §3.3)
  max_sessions = 64             # HARD admission gate — total live procs; reject AgentOverloaded (default 64; §3.3)
```

## 3. `bridge-claude` — `ClaudeCliBackend` (warm process per `SessionId`, mirroring `AcpBackend`)

New crate `crates/bridge-claude`. `ClaudeCliBackend` implements `AgentBackend`. The design **mirrors `AcpBackend` semantics so the same per-turn `forget_session`/binding eviction is harmless**, and adds a bounded warm-pool because it is process-per-session.

### 3.1 State + the critical `forget_session` semantics `[review BLOCKER fix]`
```
sessions: Arc<Mutex<HashMap<SessionId, Arc<SessionSlot>>>>   // get-or-insert a handle; proc minted in a OnceCell
SessionSlot { proc: OnceCell<Arc<SessionProc>>, last_used: AtomicInstant }
SessionProc { stdin: <writer>, turn_lock: Mutex<()>, terminated: AtomicBool,
              reader→per-turn mpsc registry, child: Supervised, claude_session_id: Option<String> }
session_cfg: Arc<Mutex<HashMap<SessionId, EffectiveConfig>>>   // configure_session stash (applied at spawn)
```
- **`forget_session(session)` DROPS ONLY THE CONFIG STASH — it does NOT kill the process** (exactly like `AcpBackend`, acp_backend.rs:1370). The 3b `BindingGuard` calls `forget_session` on every per-turn producer exit; because it only drops the stash, the warm process **survives across same-TaskId turns**, and the **in-flight-turn race is gone** (turn 1's guard-drop no longer tears down the process turn 2 uses). The process is reaped ONLY by the pool (§3.3) — never by `forget_session`.
- **`invalidate_slot(session)` — ONE primitive for all four teardown paths** (idle-TTL reap, LRU evict, cancel hard-kill, per-turn-timeout kill) `[review #7]`: under the `sessions` map lock, **remove the slot by map identity** (a `tokio::OnceCell` cannot be un-set, so the slot must be *removed/replaced*, not reset), and set the proc's `terminated` flag + `Supervised::terminate`. Each caller first holds the proc's **turn-lock** (or the cancel latch) so invalidation never races a live turn — see §3.3/§4. A subsequent `prompt` get-or-inserts a *fresh* slot → cold respawn.

### 3.2 Lifecycle (a turn)
- **`configure_session(session, eff)`** → stash `eff` (applied at the session's lazy spawn; model fixed per process — §4).
- **`prompt(session, parts)`** → get-or-insert the `SessionSlot`; `proc.get_or_try_init(spawn)` lazily spawns (exactly-once; a spawn failure leaves it unminted for retry — 3a) `claude --input-format stream-json --output-format stream-json --verbose [--model …] [perm/trust flags] [args…]` via `Supervised` (process-group + reap, 3a), bounded by a **spawn/init timeout** (§4); start the reader task; capture the init `session_id`. Acquire the **per-session turn lock**; **then REVALIDATE the proc — if `terminated` is set, the slot was reaped/cancelled out from under us in the window between cloning the proc Arc and taking the lock `[review #1 TOCTOU]`: drop this dead handle, `invalidate_slot` (idempotent), re-get-or-insert a fresh slot and respawn before proceeding** (never write to a dead stdin → never a spurious `AgentCrashed`). Then write the user envelope (§3.4); the reader demuxes events to this turn's mpsc → `Update::Text` (deltas) then `Update::Done{stop_reason}` (terminal `result`), bounded by a **per-turn timeout** (§4). Update `last_used`. **The process stays warm.** A **follow-up with the same TaskId** reuses the warm `SessionProc` (no respawn) — multi-turn continuity, same as `AcpBackend` reuses its ACP session.
  - **Why both halves are needed `[review #1]`:** `prompt` clones the proc Arc, *releases the map lock* (it cannot hold it across the `.await` on the turn-lock), then awaits the turn-lock — so map-lock alone cannot exclude a reaper. The reaper closes its side by **holding the turn-lock while it invalidates** (§3.3); `prompt` closes its side by the **post-lock `terminated` revalidation** above. Together they make "reap vs just-arriving follow-up" safe.
- **`cancel(session)`** → §4 (interrupt or kill+invalidate). **`retire`** → reap ALL warm procs (idempotent, take-once per proc). **`forget_session`** → §3.1 (stash only).

### 3.3 The bounded warm-pool (the process-per-session difference) `[review]`
Because each session is a heavyweight process (unlike `AcpBackend`'s one-process-many-sessions), bound the pool. **Two distinct bounds `[review #3]`: a hard `max_sessions` ADMISSION gate (caps total live procs, busy or idle) and a soft idle-retention pool (`idle_ttl`/`max_warm`, caps only IDLE procs).** The reaper protocol is **turn-lock-held-during-invalidation** (closes the §3.2 TOCTOU):
- **Reaper protocol (every reap/evict path) `[review #1]`:** to terminate a proc the reaper must **`try_lock` the proc's turn-lock and HOLD it** while it calls `invalidate_slot` (§3.1: remove the slot by map identity + set `terminated` + `Supervised::terminate`). `try_lock` fails ⇒ the proc is mid-turn ⇒ skip it this pass (never block, never kill mid-turn). Holding the lock across invalidation means a `prompt` that cloned the proc Arc before the reap will, on finally acquiring the lock, observe `terminated` and respawn (§3.2).
- **Idle-TTL reap:** a background reaper (interval ~60 s) selects procs where `now - last_used > idle_ttl`, then applies the reaper protocol above. **Default `idle_ttl = 55 min`** — just under the ~1 h Max prompt-cache window: held that long the session stays cache-warm; past it, a warm process ≈ `--resume` in cost (the cached prefix has expired), so reaping there is the natural break. Configurable via `extensions.idle_ttl_secs`.
- **Idle-retention cap (`max_warm`, default 16):** caps the number of **idle** procs retained for warmth. On exceeding, evict the **least-recently-used idle** session via the reaper protocol; a later follow-up re-spawns cold (continuity best-effort over the cap). This is NOT a process bound — busy procs are not counted or evicted.
- **Hard admission gate (`max_sessions`, default 64) `[review #3 — the real process bound]`:** `max_warm` is soft (it never evicts a mid-turn proc), so without a second bound an adversarial/load client spamming distinct TaskIds spawns unbounded heavyweight `claude` procs. So `prompt`'s lazy spawn is gated by a count of LIVE procs (busy + idle): at `max_sessions`, first **try to reap an idle proc** to make room; if none is idle, **reject the new session with `BridgeError::AgentOverloaded` (→ A2A error), do not spawn**. Existing sessions' follow-ups are never rejected (their proc already exists). Default 64; configurable `extensions.max_sessions`. This is the hard ceiling the soft cap lacks.
- **Retire/shutdown:** `retire()` (registry remove/edit/3b retirement) + process shutdown reap **all** procs via `invalidate_slot`/`Supervised::terminate` (lease-drained by 3b's retirement task). **Forced-retirement exception `[review #2]`:** the no-mid-turn invariant is **best-effort**; 3b's registry force-retires after a bounded grace (~30 s) **even while leases remain** (the stuck-lease backstop) and then calls `retire()`. So `retire()` first tries the turn-lock-aware path (skip+retry mid-turn procs within the grace), but on **forced** retirement after the grace it terminates remaining procs **unconditionally**, ending any still-in-flight turn (its producer sees EOF → `Failed`). This is the ONE explicit, bounded exception to "never mid-turn" — it matches 3b's existing ACP force-retire semantics and is required to honor the registry's bounded-shutdown contract.
- All *idle/LRU* reaping is **turn-lock-aware** — a proc is never terminated while a turn holds its lock; the only mid-turn termination is the bounded forced-retirement backstop above.

### 3.4 stream-json wire shapes `[probe — Task 0]`
- **Write (stdin NDJSON, per turn):** `{"type":"user","message":{"role":"user","content":[{"type":"text","text":<part text>}]}}`.
- **Read (stdout NDJSON):** init (`{"type":"system","subtype":"init","session_id":…}`); assistant text (`{"type":"assistant","message":{…}}` or, with `--verbose`/`--include-partial-messages`, incremental deltas) → `Update::Text`; terminal `{"type":"result","subtype":…,"stop_reason":…}` → `Update::Done` **on `subtype:"success"`**, or **`Err(AgentCrashed)` → `Failed` on an error subtype** (e.g. `error_max_turns`/`error_during_execution`) `[review]`. **Tolerant reader** (3a): unmodeled events / non-text content dropped. The reader, between serialized turns, ignores late/init events so they aren't misrouted into the next turn. Exact field names + whether `--verbose` is needed for deltas are pinned by Task 0.

### 3.5 Reuses proven `AcpBackend` patterns
Reader-loop → per-turn mpsc demux; per-session turn lock (one turn at a time; a second prompt for an active session waits); lazy exactly-once `OnceCell` spawn + spawn-failure retry; `Supervised` hygiene; tolerant NDJSON reader; the `BackendStream` shape (`Update::Text…` then `Update::Done`, `Err` on transport/process failure → 3b producer maps to `Failed`).

## 4. Timeouts, cancel, auth/trust, model, errors

- **Bounded timeouts (mirror `AcpBackend`) `[review]`:** a **spawn/init timeout** (the first turn's init must arrive within a bound, else `AgentNotAuthenticated`/`AgentCrashed` — a `claude` blocked on a no-TTY workspace-trust or tool-permission prompt must NOT hang `ensure` forever); a **per-turn timeout** (a turn's terminal `result` must arrive within a bound, else terminate+`Err`, so a hung turn can't hold the turn lock forever); a **cancel grace** (interrupt → wait → escalate to terminate). Constants with `AcpConfig`-style defaults; the trust/permission flags (e.g. `--permission-mode`, a trust flag) needed for a non-interactive session are pinned by Task 0.
- **Cancel `[review]`:** prefer a stream-json **interrupt control message** (if Task 0 confirms support + a terminal cancel-`result` event) → map to `Update::Done{cancelled}` → A2A `Canceled`. Fallback: `invalidate_slot` (§3.1) — `Supervised::terminate` the session's proc → reader EOF. **A hard kill must map to `Canceled`, not `Failed`** (do not let EOF-after-cancel become `AgentCrashed`) — track a per-session "cancel requested" latch (like 3a) so the EOF is interpreted as cancellation; `invalidate_slot` removes the slot so the next prompt respawns rather than using a dead handle. **Cancel-classification PRECEDENCE `[review #6]`:** when the cancel latch is set, a terminal `result` (or EOF) for that turn maps to `Canceled` **even if its subtype is error-flavored** (`error_*`) — the cancel latch takes priority over the generic §3.4 error-subtype→`Failed` rule, so a cancel that surfaces as an error subtype is never mis-reported as `Failed`.
- **Auth/trust `[review]`:** spawn plain `claude` (no `ANTHROPIC_API_KEY`) → reuse the subscription login. **Not-logged-in + workspace-trust must be DETECTABLE** (Task 0 captures the signal — stderr text? a `result` error subtype? non-zero exit?) and mapped to `BridgeError::AgentNotAuthenticated` (→ A2A `AuthRequired`), NOT a generic crash. Document the billing note (subscription Agent-SDK credit pool from 2026-06-15).
- **Model/mode:** `model` → `--model` at spawn (fixed per warm process — Claude has no mid-session switch). `configure_session`'s model applies at spawn; **a per-request model override on a follow-up to a live session is a no-op** (same mint-time-only semantic as `AcpBackend`) — documented. `mode`/permission → CLI flags (best-effort).
- **Errors:** process crash / stdout EOF mid-turn (not after a cancel) → `Err(AgentCrashed)` → `Failed`; spawn failure → `AgentCrashed`/`AgentNotAuthenticated`; unparseable NDJSON → tolerant drop; result error-subtype → `Failed` (§3.4).

## 5. Testing

- **Pattern-1 (`--resume`) — a GATED FOLLOW-ON increment, NOT co-equal `[review #5]`:** rev2 asserted Pattern-1 co-equal, but the reviews showed it is a genuinely **different state machine** with **its own copy of the rev1 blocker** that this spec does not solve: process-per-turn with no warm proc to hold continuity, so the stored `claude_session_id` **must survive the per-turn `forget_session`** (the binding evicts every turn) — otherwise `--resume` continuity breaks exactly as the warm process did in rev1. The kind-seam absorbs the *factory* swap with no registry change, but the **internals are unspecified** (where the resume-id is persisted per `SessionId` so it outlives `forget_session`; resume-failure invalidation; its own inbound e2e asserting `--resume` continuity with *different* assertions than warm). **Decision:** if Task 0 shows warm-multi-turn is impossible, Pattern-1 is designed + built as a **separate follow-on increment (3c.1)** — its minimum design: store `claude_session_id` in a **per-`SessionId` resume-stash that `forget_session` does NOT drop** (distinct from the config stash it does drop), `claude -p --output-format stream-json [--resume <id>] -- <prompt>` per turn (agent-knowledge's proven server-side-continuity pattern), invalidate the resume-stash on resume failure, cancel = kill the per-turn process, and its own inbound multi-turn e2e. - **Unit (in-process fake `claude`):** a `/bin/sh`/test-bin fake reading stdin NDJSON + emitting canned stdout NDJSON. Cover (warm): prompt → `Update::Text`×N + `Done`; `forget_session` drops the stash but **does NOT kill the proc** (assert the proc still serves a follow-up); per-session isolation; tolerant reader; result error-subtype → `Err`; idle-TTL reap (with a short TTL override) terminates an idle proc but **not one mid-turn**; `invalidate_slot` respawns a fresh proc on the next prompt (the shared primitive — one test per path: reap/LRU/cancel/timeout); LRU eviction over a small cap; **hard `max_sessions` admission gate rejects an N+1 distinct session with `AgentOverloaded` when all N are mid-turn** (#3); cancel→`Canceled` + slot invalidation; **cancel-result with an error subtype still maps to `Canceled`** (the precedence, #6); spawn-failure retry; spawn/turn timeout bounds.
- **Reaper-vs-just-arriving-follow-up race test (CRITICAL — the rev3 blocker) `[review #1]`:** deterministically interleave a reap and a follow-up `prompt` on the same idle session: arrange the prompt to clone the proc Arc, then let the reaper acquire the turn-lock + `invalidate_slot` (set `terminated`), then let the prompt acquire the (now-stale) lock — assert it **observes `terminated`, respawns a fresh proc, and the turn succeeds** (NOT a spurious `AgentCrashed`/`Failed`). Use a test seam (e.g. a barrier/`Notify`) to force the window rather than relying on timing.
- **Inbound-level sequential multi-turn e2e (CRITICAL — direct-`prompt` tests hide the blocker) `[review]`:** drive **two sequential `message/send` to the SAME TaskId THROUGH `InboundServer`**, against a fake-claude backend, and assert **turn 2 reaches the SAME warm process with prior context** (i.e. `forget_session` between turns did NOT destroy continuity). **Synchronization (must, else false-pass `[review #4]`):** `BindingGuard::drop` spawns `forget_session` *asynchronously*, so waiting only for turn-1's terminal is insufficient — turn 2 could start before cleanup runs and a (wrong) kill-the-proc impl would still pass by racing ahead. The test must **wait until the fake backend has OBSERVED `forget_session` for turn 1 (or the binding is evicted)** — e.g. the fake records a `forget_session` signal the test awaits — **before sending turn 2**. Only then does turn 2 prove the proc survived a completed eviction.
- **Wire-golden:** the outbound user envelope `{"type":"user","message":{"role":"user","content":[{"type":"text","text":…}]}}` — hand-authored expected.
- **Config/registry:** `kind="claude-cli"` parses (+ default `acp`); `kind` in the reuse-identity tuple (an `acp`↔`claude-cli` edit forces a fresh slot); invalid kind → `ConfigError::Registry`; the kind-dispatching factory builds the right backend.

## 6. Scope boundary
**3c BUILDS:** the `AgentKind` discriminator (+ config parse + `kind` in reuse-identity) + the kind-dispatching factory; `bridge-claude`/`ClaudeCliBackend` **warm Pattern 2** with the bounded warm-pool — `forget_session`-drops-stash, the `invalidate_slot` primitive (reaper-holds-turn-lock + post-lock `terminated` revalidation), idle-TTL/`max_warm` idle-retention + the hard `max_sessions` admission gate, retire/forced-retire, bounded timeouts, cancel→Canceled (+ precedence), auth/trust detection; the inbound-level multi-turn e2e (synchronized on observed `forget_session`) + the reaper-race test + the gated real e2e. **3c DOCUMENTS / designs-in (not built):** **Pattern-1 (`--resume`) as a gated follow-on increment 3c.1** (built only if Task 0 shows warm-multi-turn is impossible — §5); the `claude-api` (B1) HTTP arm + the exec-residue (`cmd`/`allowed_cmds`) it will force changing — named for the **post-3c conductor re-evaluation**; the ACP-Claude adapter (A) as a `kind="acp"` config entry. **Non-goals:** no Anthropic API/HTTP code, no API key, no fan-out/registry changes beyond the `kind` field + reuse-identity; Pattern-1 internals are out of 3c proper.
