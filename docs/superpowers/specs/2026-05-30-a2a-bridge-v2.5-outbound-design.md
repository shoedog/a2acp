# A2A Bridge — Increment 2.5 Design Spec (Outbound Delegation, Passthrough)

*Date: 2026-05-30*
*Status: Approved design — ready for implementation planning*
*Builds on: v1 (`docs/superpowers/specs/2026-05-29-a2a-bridge-v1-design.md`, merged to `main`)*

---

## 1. Purpose

Implement the concrete **outbound A2A path** deferred from v1: a single inbound A2A task is
forwarded wholesale to **one configured remote A2A peer**, and the peer's streamed events
flow back to the caller's SSE. This realizes v1 success-criterion **S2** (a delegated
outbound call round-trips into the caller's stream).

This is **passthrough** delegation — Kiro is not involved on this path and there is no
concurrent stream-merge. The fan-out + two-producer merge (Kiro *and* peer, merged) is a
separate, named **Increment 2.6**.

## 2. Decisions (locked)

| Decision | Choice |
|----------|--------|
| Topology | Passthrough: one inbound task → one configured remote peer; no merge (merge → 2.6) |
| Outbound client | **Hand-rolled on `reqwest`** using `a2a-lf` wire types (mirrors how inbound/ACP were built) |
| Peer | One static peer in TOML config (URL + bearer auth); no discovery/registry (mesh deferred) |
| Outbound auth | A configured bearer token the bridge presents to the peer; inbound caller identity is NOT forwarded in 2.5 |
| Routing | `RouteDecision` returns `RouteTarget { Local(AgentId), Delegate }`; **skill selected via `message.metadata["a2a-bridge.skill"]`**, parsed into `TaskMeta.skill: Option<String>` `[Codex 1 / Claude]` |
| Message content | Flesh out `Part` to carry text; thread real content to the peer AND (bonus fix) to Kiro |
| Outbound stream binding | Parse each peer SSE `data:` as **`a2a::StreamResponse`** (`task`/`message`/`statusUpdate`/`artifactUpdate`), not our inbound event names; final `Artifact` only on `artifactUpdate.lastChunk=true` or terminal `TaskState` `[Codex 2]` |
| Completion vs EOF | Completion = A2A terminal `TaskState`/final artifact chunk. A bare SSE EOF with no terminal signal → `UpstreamA2aError`, never synthesized success `[Codex 3]` |
| `DelegationPort` shape | `delegate(auth, local_task_id, parts)` returns a stream **plus** a peer-task-id `watch` channel + a `cancel()` path; store records `local_task → peer_task` `[Codex 4 / Claude]` |
| Merge point | Reuse the existing inbound `mpsc<Result<Event,BridgeError>>` → SSE path; delegation is one alternative producer (2.6's merge is more than a second producer — see §8) |

## 3. Scope

### 3.1 In scope
1. **Streaming `DelegationPort`** in `bridge-core::ports` — yields a stream of
   `translator::Event` (same type the Kiro path emits), so it plugs into the existing
   `mpsc → SSE` path with no new SSE plumbing.
2. **Outbound A2A client** (`bridge-a2a-outbound`): `reqwest` POSTs a `SendStreamingMessage`
   JSON-RPC body (built from `a2a-lf` types, carrying the forwarded `Vec<Part>`), reads the
   peer's SSE response, maps peer `status-update`/`artifact-update` frames → our `Event`.
3. **Peer config** (TOML `[delegation]`): `peer_url`, `auth` (env-expanded bearer).
4. **Message-content threading**: `Part { text: String }`; inbound parsing produces real
   `Vec<Part>`; forwarded to the peer body and to `KiroBackend` (fixes the v1 "content not
   threaded" limitation on the local path too).
5. **Routing**: `RouteTarget` enum; two-skill Agent Card (`kiro-code`, `delegate`); the
   inbound `gate()` branches Local vs Delegate.
6. **Cancellation**: inbound `CancelTask` on a delegated task → `CancelTask` to the peer
   (using the peer's returned task id) + abort the local producer.
7. **Errors/lifecycle**: peer unreachable / non-2xx / malformed SSE → `UpstreamA2aError` →
   task `Failed`; configurable outbound timeout; tolerant reader on peer frames.

### 3.2 Out of scope (→ later)
Fan-out (Kiro + peer concurrently) and the two-producer concurrent stream-merge
(**Increment 2.6**); multiple peers; discovery/registry/mesh; inbound→outbound identity
propagation; file/image `Part` content (text only in 2.5); MCP-triggered delegation.

### 3.3 Success criteria
- **S2 (realized).** A `delegate`-skill `SendStreamingMessage` whose message text is `X`
  causes the bridge to POST a `SendStreamingMessage` carrying `X` to the configured peer,
  and the peer's artifact streams back as the caller's SSE `artifact-update`.
- **S2a (content threading).** The peer (and Kiro, on the local path) receive the caller's
  actual message text — asserted by a mock peer inspecting the request body.
- **S2b (cancel).** An inbound `CancelTask` on a delegated task causes a `CancelTask` to the
  peer for the peer's task id.
- **S2c (failure).** A peer error surfaces as `UpstreamA2aError` → task `Failed`, with no
  silent stream close (same discipline as the v1 kiro terminal-signal fix).

                                   │  (bridge-a2a-outbound, reqwest)
              ──POST SendStreamingMessage(parts)──▶ remote A2A peer
              ◀──────────── SSE status/artifact ───

## 5. Data flow & semantics

- **Skill selection:** the inbound `message.metadata["a2a-bridge.skill"]` value is parsed into
  `TaskMeta.skill`; `RouteDecision` maps `"delegate"` → `RouteTarget::Delegate`, anything else
  → `Local(kiro)`. Absent metadata defaults to `Local` `[Codex 1]`.
- **Forward:** inbound `Vec<Part>` (text) → peer `SendStreamingMessage` message parts (via
  `a2a-lf` types). The peer's response is an A2A task streamed over SSE.
- **Map (`a2a::StreamResponse`):** parse each peer SSE `data:` as `a2a::StreamResponse`
  regardless of the SSE `event:` name `[Codex 2]`:
  - `Task` / `Message` / `StatusUpdate` → `Event::status(...)`; capture the peer task id from
    whichever variant carries it (write it into the `watch` channel).
  - `ArtifactUpdate` → `Event::status(...)` for intermediate chunks; **only `lastChunk=true`
    (or a terminal `TaskState`) yields the final `Event::artifact(...)`**.
  - Unknown/unrecognized variants skipped (tolerant reader).
- **Completion vs EOF** `[Codex 3]`: completion is signalled by a terminal `TaskState`
  (`Completed`/`Failed`/`Canceled`/`Rejected`) or the final artifact chunk — **not** by the
  socket closing. A clean SSE EOF *before* such a signal is `Err(UpstreamA2aError)`, never a
  synthesized success. (This corrects an earlier draft that treated clean EOF as completion.)
- **Cancel** `[Codex 4]`: the peer task id may not exist until the first frame, and an inbound
  `CancelTask` may arrive first. The store latches an early cancel; once the `watch` channel
  yields the peer id, the client POSTs `CancelTask{peer_id}` and aborts the producer. If the
  task ends before any peer id is observed, cancel is a no-op (nothing to cancel upstream).
- **Errors:** connect/timeout/non-2xx/SSE-parse/premature-EOF → `Err(UpstreamA2aError)`; maps
  via `disposition()` → task `Failed`.

## 6. Error model

Reuse `BridgeError`. `UpstreamA2aError` is the outbound failure surface (already in the enum,
already `→ Failed`). Add nothing unless a distinct `OutboundTimeout`/`PeerUnreachable`
granularity proves useful in implementation — default to `UpstreamA2aError` (YAGNI).

## 7. Testing

- **Mock peer** (in-test `axum` server, or `wiremock`) answering `SendStreamingMessage` with a
  canned SSE stream of real `a2a::StreamResponse` frames (`statusUpdate`, then an
  `artifactUpdate` with `lastChunk=true`). Drive the `PeerDelegation` client and assert:
  (a) the forwarded request body contains the caller's text (content-threading proof, S2a);
  (b) frames → `Event::status`/`Event::artifact` with the final artifact only on `lastChunk`
  (S2); (c) inbound cancel → mock receives `CancelTask` with the peer task id (S2b); (d) peer
  5xx / dropped stream / **clean EOF before a terminal signal** → `UpstreamA2aError` → `Failed`
  (S2c + Codex 3); (e) **early-cancel before the peer id is observed** is latched and is a
  no-op upstream (Codex 4).
- **Wired inbound→delegate integration test** `[Claude]`: drive the real `InboundServer`
  (with the 6th `DelegationPort` port = `PeerDelegation` pointing at the mock peer) via a
  `SendStreamingMessage` carrying `metadata["a2a-bridge.skill"]="delegate"`, and assert the
  caller's SSE shows the peer's artifact — proving routing + the 6th-port wiring, not just the
  `DelegationPort` in isolation.
- **Caller-disconnect test** `[Claude]`: drop the inbound SSE mid-stream → the mock peer
  receives a `CancelTask` (no leaked upstream stream).
- **Kiro-path content test**: assert `KiroBackend` now sends the real parts in `session/prompt`
  (regression-proofs the bonus fix; scripted child echoes the received prompt).
- **Gated `#[ignore]` real e2e**: one `a2a-bridge` instance's `delegate` skill → a second
  instance's `kiro-code` skill (bridge-to-bridge full loop), run manually.
- Coverage gates unchanged (workspace ≥85%; bridge-core/bridge-acp ≥90%); the new outbound
  client crate aims for high line coverage via the mock peer.

## 8. Forward note — Increment 2.6 (fan-out + merge)

2.5 deliberately leaves the `mpsc → SSE` path single-producer. 2.6 is **not just "a second
producer"** `[Claude]` — it adds a dedicated merge stage with real semantics: a `fan-out` skill
that drives **both** Kiro and the peer; a merge policy for ordering and **per-source event
labeling** (so the caller can tell Kiro's output from the peer's); **done-when-both-finish**
termination; and a **partial-failure policy** (one source fails — fail the task, or degrade to
the survivor?). 2.5's streaming `DelegationPort`, peer-id `watch`/cancel handle, and reqwest
client are the building blocks 2.6 composes, but the merge stage itself is the substance of
2.6 and gets its own design pass.
