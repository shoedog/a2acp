# A2A Bridge — Increment 2.5 Design Spec (Outbound Delegation, Passthrough)

*Date: 2026-05-30*
*Status: Approved design — ready for implementation planning*
*Builds on: v1 (`docs/superpowers/specs/2026-05-29-a2a-bridge-v1-design.md`, merged to `main`)*

---

## 1. Purpose

Implement the concrete **outbound A2A path** deferred from v1: a single inbound A2A task is
forwarded wholesale to **one configured remote A2A peer**, and the peer's streamed events
flow back to the caller's SSE. This realizes v1 success-criterion **S2** (a delegated
outbound call round-trips into the caller's stream).

This is **passthrough** delegation — Kiro is not involved on this path and there is no
concurrent stream-merge. The fan-out + two-producer merge (Kiro *and* peer, merged) is a
separate, named **Increment 2.6**.

## 2. Decisions (locked)

| Decision | Choice |
|----------|--------|
| Topology | Passthrough: one inbound task → one configured remote peer; no merge (merge → 2.6) |
| Outbound client | **Hand-rolled on `reqwest`** using `a2a-lf` wire types (mirrors how inbound/ACP were built) |
| Peer | One static peer in TOML config (URL + bearer auth); no discovery/registry (mesh deferred) |
| Outbound auth | A configured bearer token the bridge presents to the peer; inbound caller identity is NOT forwarded in 2.5 |
| Routing | `RouteDecision` returns `RouteTarget { Local(AgentId), Delegate }`; Agent Card advertises two skills |
| Message content | Flesh out `Part` to carry text; thread real content to the peer AND (bonus fix) to Kiro |
| Merge point | Reuse the existing inbound `mpsc<Result<Event,BridgeError>>` → SSE path; delegation is one alternative producer |

## 3. Scope

### 3.1 In scope
1. **Streaming `DelegationPort`** in `bridge-core::ports` — yields a stream of
   `translator::Event` (same type the Kiro path emits), so it plugs into the existing
   `mpsc → SSE` path with no new SSE plumbing.
2. **Outbound A2A client** (`bridge-a2a-outbound`): `reqwest` POSTs a `SendStreamingMessage`
   JSON-RPC body (built from `a2a-lf` types, carrying the forwarded `Vec<Part>`), reads the
   peer's SSE response, maps peer `status-update`/`artifact-update` frames → our `Event`.
3. **Peer config** (TOML `[delegation]`): `peer_url`, `auth` (env-expanded bearer).
4. **Message-content threading**: `Part { text: String }`; inbound parsing produces real
   `Vec<Part>`; forwarded to the peer body and to `KiroBackend` (fixes the v1 "content not
   threaded" limitation on the local path too).
5. **Routing**: `RouteTarget` enum; two-skill Agent Card (`kiro-code`, `delegate`); the
   inbound `gate()` branches Local vs Delegate.
6. **Cancellation**: inbound `CancelTask` on a delegated task → `CancelTask` to the peer
   (using the peer's returned task id) + abort the local producer.
7. **Errors/lifecycle**: peer unreachable / non-2xx / malformed SSE → `UpstreamA2aError` →
   task `Failed`; configurable outbound timeout; tolerant reader on peer frames.

### 3.2 Out of scope (→ later)
Fan-out (Kiro + peer concurrently) and the two-producer concurrent stream-merge
(**Increment 2.6**); multiple peers; discovery/registry/mesh; inbound→outbound identity
propagation; file/image `Part` content (text only in 2.5); MCP-triggered delegation.

### 3.3 Success criteria
- **S2 (realized).** A `delegate`-skill `SendStreamingMessage` whose message text is `X`
  causes the bridge to POST a `SendStreamingMessage` carrying `X` to the configured peer,
  and the peer's artifact streams back as the caller's SSE `artifact-update`.
- **S2a (content threading).** The peer (and Kiro, on the local path) receive the caller's
  actual message text — asserted by a mock peer inspecting the request body.
- **S2b (cancel).** An inbound `CancelTask` on a delegated task causes a `CancelTask` to the
  peer for the peer's task id.
- **S2c (failure).** A peer error surfaces as `UpstreamA2aError` → task `Failed`, with no
  silent stream close (same discipline as the v1 kiro terminal-signal fix).

## 4. Architecture & component changes

```
caller ──SendStreamingMessage(skill=delegate)──▶ bridge-a2a-inbound
   gate(): auth → version → route=Delegate
                                   │
                                   ▼
                         DelegationPort.delegate(parts)
                                   │  (bridge-a2a-outbound, reqwest)
              ──POST SendStreamingMessage(parts)──▶ remote A2A peer
              ◀──────────── SSE status/artifact ───
                                   │ map peer frames → Event
                                   ▼
                         mpsc<Result<Event,_>> ──▶ SSE (existing path)
```

| File | Change |
|------|--------|
| `bridge-core/src/ports.rs` | `DelegationPort::delegate(&self, parts: Vec<Part>) -> Result<DelegationStream, BridgeError>` where `DelegationStream` yields `Result<Event, BridgeError>`. `RouteDecision::route -> RouteTarget`. |
| `bridge-core/src/domain.rs` | `Part { text: String }`; `RouteTarget { Local(AgentId), Delegate }`. |
| `bridge-core/src/translator.rs` | `Event`/`EventKind` reused by the delegation path (no change, or expose a small constructor). |
| `bridge-a2a-outbound/src/client.rs` | New reqwest A2A client: build `SendStreamingMessage` from `a2a-lf` types, read SSE, map frames → `Event`; capture the peer task id; `cancel(peer_task_id)`. |
| `bridge-a2a-outbound/src/lib.rs` | `PeerDelegation` impl of `DelegationPort` (replaces `StubDelegation`). |
| `bridge-a2a-inbound/src/server.rs` | `gate()` returns/branches on `RouteTarget`; Delegate path runs the `DelegationPort` producer into the existing mpsc→SSE; `CancelTask` routes to peer if delegated. Two-skill card. |
| `bridge-a2a-inbound/src/card.rs` | Advertise `kiro-code` + `delegate` skills. |
| `bridge-acp/src/kiro.rs` | Serialize the real `Vec<Part>` text into the `session/prompt` request (was empty). |
| `bin/.../config.rs` + `main.rs` | `[delegation]` config; wire `PeerDelegation` (not `StubDelegation`); `RouteDecision` that maps the `delegate` skill → `Delegate`. |
| `bridge-store` | Record delegated task → peer task id (for cancel). |

## 5. Data flow & semantics

- **Forward:** inbound `Vec<Part>` (text) → peer `SendStreamingMessage.params.message.parts`.
  The peer's response is an A2A task; the bridge reads its SSE stream.
- **Map:** peer `status-update` → `Event{Status}`; peer `artifact-update` → `Event{Artifact}`
  (final). Unknown peer frames skipped (tolerant reader). Stream ends on the peer's terminal
  frame. The v1 terminal-signal rule applies — **never a silent close**: a *clean* SSE EOF
  with no terminal frame synthesizes a terminal `Event{Artifact}` (treat as completion); a
  *transport error* mid-stream (connection drop, parse failure) is `Err(UpstreamA2aError)`.
- **Cancel:** capture the peer's task id from its first response; on inbound `CancelTask`,
  POST `CancelTask{peer_task_id}` to the peer and drop the local producer.
- **Errors:** connect/timeout/non-2xx/SSE-parse → `Err(UpstreamA2aError)`; maps via
  `disposition()` → task `Failed`.

## 6. Error model

Reuse `BridgeError`. `UpstreamA2aError` is the outbound failure surface (already in the enum,
already `→ Failed`). Add nothing unless a distinct `OutboundTimeout`/`PeerUnreachable`
granularity proves useful in implementation — default to `UpstreamA2aError` (YAGNI).

## 7. Testing

- **Mock peer** (in-test `axum` server or `wiremock`) answering `SendStreamingMessage` with a
  canned SSE stream. Assert: (a) forwarded request body contains the caller's text
  (content-threading proof, S2a); (b) peer frames → caller SSE artifact (S2); (c) inbound
  cancel → mock receives `CancelTask` with the peer task id (S2b); (d) peer 5xx / dropped
  stream → `UpstreamA2aError` → task `Failed` (S2c).
- **Kiro-path content test**: assert `KiroBackend` now sends the real parts in `session/prompt`
  (regression-proofs the bonus fix; scripted child echoes the received prompt).
- **Gated `#[ignore]` real e2e**: one `a2a-bridge` instance's `delegate` skill → a second
  instance's `kiro-code` skill (bridge-to-bridge full loop), run manually.
- Coverage gates unchanged (workspace ≥85%; bridge-core/bridge-acp ≥90%); the new outbound
  client crate aims for high line coverage via the mock peer.

## 8. Forward note — Increment 2.6 (fan-out + merge)

2.5 deliberately leaves the `mpsc → SSE` path single-producer. 2.6 adds a `fan-out` skill
where a task drives **both** Kiro and the peer, both producers feeding the same channel, with:
done-when-both-finish, per-source event labeling, and a partial-failure policy. 2.5's
streaming `DelegationPort` + reqwest client are the building blocks 2.6 composes.
