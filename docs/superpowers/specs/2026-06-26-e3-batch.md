# E3 — Parallel Batch Dispatch — SPEC

**One-liner:** `a2a-bridge run-batch <workflow> --manifest <file>` submits N **independent** workflow runs to a
running `serve`, runs them concurrently under a **real, serve-wide concurrency cap**, durably — each child is a
normal detached task (W3b-resumable, E1-worktree-isolated, individually cancelable), with a durable parent `batch`
record giving one-query status, cancel-the-batch, and crash-safe tail-resume.

**Roadmap:** Slice-10+ tail item E3 (the user picked it from {E3 batch · E7 typed task-spec · E8 prompt-lib}).
E3 = "Run N **independent** tasks concurrently across the A1 pool (concurrency cap), like a parallel/pipeline
primitive — for fan-out **implementation, not just panels**" (`docs/orchestration-improvements-2026-06-17.md:103`).
Base = `main` `80be1b4`. Branch `feat/e3-batch`.

---

## Goal & value
Fan-out **within** one workflow already works (the executor's `FuturesUnordered` pool + the Slice-10 `panel` →
synth). E3 is the **orthogonal axis**: N *independent* top-level jobs. The operator's real need is "implement these
12 tickets, 4 at a time, each in its own worktree" or "code-review every PR in this list, 6 at a time" — today that
is 12 hand-managed `run-workflow`/`message/send` invocations with no shared cap, no roll-up, and no crash-safety for
the batch as a whole.

E3 layers a **bounded-concurrency batch** over the existing per-job detached primitive:
- each child is submitted through the **existing** `spawn_detached_workflow` path → W3b crash-resume + E1 worktree
  isolation + `task get/cancel` per child come for FREE;
- a **serve-wide semaphore** caps concurrency across *all* batches (two batches can't blow past the pool — the
  property a per-CLI cap can't give);
- a durable parent `batch` record makes "batch status" one query, makes `cancel_batch` atomic, and makes the
  **un-submitted tail** survive a serve restart.

Net: one command runs a whole work-list to completion, bounded, durable, isolated — the substrate for self-hosting
the implement/review loop at list scale.

## Scope (MVP cut-line)

**IN:**
1. **`run-batch` CLI** (`a2a-bridge run-batch <workflow> --manifest <file> [--concurrency K] [--detach]`) — a thin
   A2A client of a running `serve`. Submits, prints the `BatchId`, then tails a live roll-up (polling
   `batch_status`) until terminal; **Ctrl-C detaches** (the batch keeps running serve-side); `--detach` skips the
   tail and just prints the id.
2. **`batch` CLI** (`batch status <id>` / `batch list` / `batch cancel <id>`).
3. **`BatchId` newtype** + `new_batch_id()` (mirrors `new_detached_task_id`).
4. **Durable parent** — a `batch` table `(id, workflow, concurrency, total, status, items_json, created_ms,
   updated_ms)`; `TaskRecord` gains `batch_id: Option<BatchId>` + `item_id: Option<String>`. Additive SQLite
   migration (W3a/W3b additive-migration precedent).
5. **Serve-side admission loop + serve-wide cap** — `Coordinator::run_batch` spawns a per-batch admission task that
   admits children as slots free; every child-admit first acquires a permit from **one serve-wide `Semaphore`**
   (`Arc<Semaphore>` on the Coordinator, sized by `[batch] max_concurrent`), released on child completion → the cap
   is **global across batches**. Each batch's `--concurrency K` is a per-batch sub-limit (≤ global).
6. **Coordinator service API** (Slice-8 ruling: Coordinator IS the service; A2A/CLI/MCP are thin adapters) —
   `run_batch(BatchParams) -> BatchId`, `batch_status(BatchId) -> BatchStatus`, `batch_list(limit) -> Vec<BatchSummary>`,
   `cancel_batch(BatchId) -> bool`, exposed as four new JSON-RPC method arms on the inbound server
   (`crates/bridge-a2a-inbound/src/server.rs:713`).
7. **Continue-others failure policy** — one child failing NEVER aborts the batch; the batch settles `Completed` once
   all children are terminal, carrying `ok=N / failed=M`.
8. **Cancel-the-batch** — `cancel_batch` → batch `Cancelling`: stop admitting + cancel in-flight children via the
   existing `workflow_cancels` tokens; settles `Cancelled`.
9. **Crash-safe tail-resume** — on serve boot, a new `resume_working_batches` re-reads each `Working` batch,
   re-drives its admission loop for the **un-submitted tail** + still-`Working` children, **all through the
   serve-wide semaphore** (so the global cap holds across a restart too).

**OUT (explicit deferrals):**
- **Global admission over NON-batch traffic** — the serve-wide semaphore gates **batch children only**; a single
  `run_workflow`/`message/send` still spawns immediately (unchanged). Routing *all* detached traffic through global
  admission is a larger change to the existing spawn-immediately path → DEFER.
- **`--fail-fast`** — only continue-others in the MVP.
- **Glob / directory input expansion** — manifest-only first (a glob would expand into a manifest client-side later).
- **Per-item different workflows** — one workflow per batch.
- **Batch-of-batches / nested batches.**
- **Live SSE batch stream** — the MVP polls `batch_status`; a sequenced batch event stream (cf. Slice-15 streaming
  reattach for tasks) is a follow-on.
- **Per-batch retry / re-run-the-batch** — E6 already covers per-node retry *within* each child; re-running a whole
  failed batch is a follow-on (re-submit the remaining manifest items).
- **MCP `run_batch` tool** — the coordinator API is MCP-ready, but wiring the `bridge-mcp` adapter is a follow-on
  (A2A/CLI adapters only in this slice).

## Data model

```rust
// ids.rs
id_newtype!(BatchId);

// task_store.rs
pub enum BatchStatus { Working, Completed, Cancelling, Cancelled, Failed }

pub struct BatchRecord {
    pub id: BatchId,
    pub workflow: String,
    pub concurrency: u32,          // per-batch sub-limit (already clamped ≤ global at submit)
    pub total: u32,               // items.len()
    pub status: BatchStatus,
    pub items_json: String,       // serialized Vec<BatchItem> — the durable plan (for tail-resume)
    pub created_ms: i64,
    pub updated_ms: i64,
}

pub struct BatchItem {           // one manifest entry
    pub item_id: String,         // operator-supplied or index-derived; unique within the batch
    pub input: String,           // the workflow input for this child
    pub session_cwd: Option<String>,
}

pub struct BatchSummary {        // roll-up DTO (batch_status / batch_list)
    pub id: BatchId,
    pub workflow: String,
    pub status: BatchStatus,
    pub total: u32,
    pub ok: u32,                 // children Completed
    pub failed: u32,             // children Failed/Interrupted
    pub running: u32,            // children Working
    pub pending: u32,            // items with no child row yet (un-submitted tail)
    pub children: Vec<(String /*item_id*/, TaskId, TaskRecordStatus)>,
}

// TaskRecord gains:
//   pub batch_id: Option<BatchId>,
//   pub item_id: Option<String>,
```

`BatchStatus` transitions: `Working → Completed` (all children terminal, none mid-flight) | `Working → Cancelling →
Cancelled` (cancel) | `Working → Failed` (submit-time only — malformed batch, *before* any child; per-child failures
do NOT fail the batch).

## The admission loop (the core — `crates/bridge-coordinator/src/batch.rs`)

`run_batch(BatchParams)`:
1. Validate: workflow exists (`self.workflows.get`), manifest non-empty, each item's `session_cwd` passes
   `validate_cwd(allowed_cwd_root)` (reuse `OpParams::validate_cwd`), `item_id`s unique. Reject → `Failed` at submit
   (never persisted as Working) with `InvalidRequest`.
2. `concurrency = min(params.concurrency.unwrap_or(default_concurrency), max_concurrent)`.
3. Persist the `BatchRecord` (`Working`, `items_json` = the full plan) via `create_batch`. Mint `BatchId =
   new_batch_id()`.
4. Spawn the admission task (`tokio::spawn(run_batch_admission(...))`); return the `BatchId` immediately.

`run_batch_admission(batch_id, items, concurrency, ...)` — completion-driven, mirrors W3b's `FuturesUnordered`
drain:
- A pending work-list = items with no existing child row (on a fresh submit, all of them).
- A `FuturesUnordered<JoinHandle<...>>` of in-flight children, bounded by `concurrency`.
- **Admit** an item: `let permit = self.batch_semaphore.clone().acquire_owned().await` (the **serve-wide** cap),
  then `create` a child `TaskRecord` (status `Working`, `batch_id=Some`, `item_id=Some`, `workflow_spec_json` set
  exactly like `run_workflow` so the child is W3b-resumable on its own), then `spawn_detached_workflow(...)` and push
  its `JoinHandle` (carrying the `OwnedSemaphorePermit` so the permit drops on completion) into the
  `FuturesUnordered`.
- **On a child completion** (the `JoinHandle` resolves): the permit drops (frees a global slot); admit the next
  pending item if any.
- **Cancellation** of the batch (a batch-level `CancellationToken`): stop admitting; cancel in-flight children via
  their `workflow_cancels` tokens; drain the `FuturesUnordered`.
- **Terminal**: when the work-list is empty AND the `FuturesUnordered` is drained → `set_batch_status(Completed)`
  (or `Cancelled` if canceled). The per-child ok/failed counts are derivable from the child rows at
  `batch_status`-query time (the loop need not track them).

**Why a `FuturesUnordered` AND a semaphore:** the `FuturesUnordered` gives per-batch flow control (≤ `concurrency`
of *this* batch's children in flight) and the completion signal that drives admission; the **shared semaphore** is
what makes the cap **global across batches** (the whole point of "serve-side cap"). A single batch is bounded by
`min(its concurrency, available global permits)`.

## Crash-resume — the division of labor

The global cap MUST hold across a serve restart, so batch children are **never** resumed by the plain W3b path:

- **`resume_working_tasks` (existing, `detached.rs:1423`)** resumes only **non-batch** tasks (`batch_id IS NULL`).
  It spawns them immediately as today (single runs are not capped — the documented deferral).
- **`resume_working_batches` (NEW)**, called AFTER, for each `Working` batch:
  - parses `items_json`, queries existing `batch_children`, and re-drives `run_batch_admission` with the pending
    work-list = items whose `item_id` has no child row (the **un-submitted tail**) and a `FuturesUnordered`
    re-seeded for still-`Working` children (re-run via `run_from` with their checkpoints, exactly like
    `resume_working_tasks` does, but **through the semaphore**).
  - Already-terminal children (Completed/Failed/Cancelled) are left as-is and counted in the roll-up.

Idempotency keystones: a child is keyed by `(batch_id, item_id)`; admission checks "does a child row already exist
for this item_id" before creating, so a crash between create-row and spawn (the row exists, `Working`, with a spec)
is recovered by re-spawning that item (its existing row is reused/updated, not duplicated). The `resume_attempts`
cap on each child remains the per-child poison backstop; the batch itself adds no new unbounded loop (the work-list
is finite and strictly shrinks).

## Transport & API surface

Per the Slice-8 ruling, the **Coordinator is the service** and A2A/CLI/(MCP) are thin adapters:

- **Coordinator** (`coordinator.rs`): `run_batch`, `batch_status`, `batch_list`, `cancel_batch`.
- **Inbound A2A** (`server.rs:713`): `"RunBatch"` / `"BatchStatus"` / `"BatchList"` / `"CancelBatch"` arms, each
  parsing params → calling the Coordinator → JSON-RPC result. `RunBatch` params = `{ workflow, concurrency?, items:
  [{item_id?, input|input_file-already-resolved, session_cwd?}] }` (the CLI resolves `input_file` → `input` and
  `item_id` defaults before sending, so the wire carries fully-resolved items).
- **CLI** (`main.rs`): `run-batch` reads + validates the manifest, POSTs `RunBatch` to the serve URL (default
  `http://127.0.0.1:8080`, mirroring `run-workflow --serve`), prints the `BatchId`, then polls `BatchStatus` on an
  interval rendering a roll-up until terminal (Ctrl-C → stop polling, batch keeps running). `batch
  status/list/cancel` are one-shot RPCs.

**`run-batch` requires a running `serve`** — there is no one-shot batch mode (that was the trade accepted by choosing
serve-side cap + durable; a one-shot would inherently be CLI-side + ephemeral).

## Manifest format (TOML)

```toml
# tasks.toml — one workflow for the whole batch (the <workflow> CLI arg)
[[item]]
id = "ticket-101"            # optional; defaults to the 0-based index as a string
input = "Implement the foo endpoint per docs/foo.md"
session_cwd = "/repos/foo"   # optional; per-item isolation target (validated vs allowed_cwd_root)

[[item]]
id = "ticket-102"
input_file = "tasks/102.md"  # alternative to inline input (CLI reads + inlines before sending)
```

`id` collisions, empty manifest, both/neither of `input`/`input_file`, or a `session_cwd` outside
`allowed_cwd_root` are rejected client-side (and re-validated server-side).

## Config

```toml
[batch]
max_concurrent = 6        # serve-wide cap (the shared Semaphore); REQUIRED to enable batch
default_concurrency = 4   # per-batch default when run-batch --concurrency is omitted
```

Absent `[batch]` → `run_batch` returns `InvalidRequest { field: "batch (not configured)" }` (batch is opt-in, like
`[worktrees]`). `default_concurrency` clamps to ≤ `max_concurrent` at load.

## Observability
- `tracing` spans/events: batch submit (`batch_id, workflow, total, concurrency`), each child admit (`batch_id,
  item_id, task`), each child completion (`batch_id, item_id, status`), batch terminal (`batch_id, ok, failed`).
- `batch_status` is the queryable roll-up. A rich sequenced batch event stream is deferred (OUT).

## Open questions / decisions for the reviewers

**Decisions taken (challenge if wrong):**
- **D1 — Batch unit = N workflow runs** (not an in-graph map node, not impl-only). Chosen: most general, lowest new
  foundation, subsumes `implement`, matches "independent tasks / not just panels."
- **D2 — Serve-side cap + durable parent** (not CLI-side/ephemeral). The cap must be a *real* bound on the pool.
- **D3 — The serve-wide semaphore gates BATCH CHILDREN ONLY** for the MVP; non-batch single runs stay immediate.
  (Global-over-everything = OUT.)
- **D4 — Durable plan = `items_json` on the `BatchRecord` + create-child-on-admit** (NOT pre-creating all children
  as a new `Pending` task status). Rationale: keeps the `TaskRecordStatus` enum + `resume_working_tasks` /
  `sweep_interrupted` / `list` semantics UNTOUCHED; the batch record self-describes its plan; tail = items minus
  existing child `item_id`s. *Challenge:* is a pre-created-`Pending`-children model cleaner despite the wider task-
  lifecycle blast radius?
- **D5 — Continue-others** failure policy; the batch only `Failed`s at submit (malformed). Per-child failures →
  roll-up counts.
- **D6 — Batch children excluded from the plain resume path** (`resume_working_tasks` filters `batch_id IS NULL`;
  `resume_working_batches` resumes children through the semaphore) so the global cap holds across a restart.
  *Challenge:* is the `working_tasks()`-returns-non-batch-only realization vs a call-site filter the right split?
- **D7 — `run-batch` requires `serve`** (no one-shot mode).
- **D8 — Tail-resume is IN** (per the "durable" choice), not deferred.

**Questions to resolve before/at planning:**
- **Q1 — Terminal-status ownership.** The admission loop sets `Completed`. If the loop's process dies *after* the
  last child finished but *before* it wrote `Completed`, the batch is stuck `Working` with all-terminal children
  until the next serve boot re-runs `resume_working_batches` (which finds an empty work-list + all-terminal children
  → settles `Completed`). Is that boot-driven settle sufficient, or does `batch_status` also need a **lazy settle**
  (compute "effectively complete" when queried)? (Recommend: boot-settle only; document the window.)
- **Q2 — Permit acquisition vs cancel ordering.** A child waiting on `batch_semaphore.acquire_owned()` when the
  batch is canceled must abort the acquire promptly (`tokio::select!` cancel vs acquire) and NOT create a child row.
  Confirm the admission loop's cancel arm precedes row-creation.
- **Q3 — Child cancel plumbing on restart.** After a restart, in-flight children resumed by `resume_working_batches`
  need fresh `workflow_cancels` tokens registered so a later `cancel_batch` can reach them. Confirm the resume path
  registers tokens (it must, exactly as `run_workflow` does).
- **Q4 — `item_id` charset/uniqueness.** `item_id` becomes a `TaskRecord` column and a roll-up key. Is a plain
  non-empty string fine, or should it be the strict `[a-z0-9_-]+` charset (it is not interpolated into templates —
  recommend plain, dedup-checked)?
- **Q5 — Semaphore sizing + starvation.** With `max_concurrent = 6` and three batches each `concurrency = 4`, the
  six permits are shared FIFO-ish by tokio's `Semaphore`. Is plain `Semaphore` fairness adequate, or is there a
  starvation risk (a long-running batch's children hogging permits)? (Recommend: accept `Semaphore` fairness for
  the MVP; note it.)
- **Q6 — `cancel_task` on a batch child.** Should canceling an individual child via the existing `cancel_task` be
  allowed (it would, since children are normal tasks) and how does the batch loop react (treat as a failed/canceled
  child, admit next)? Confirm it doesn't wedge the loop.
- **Q7 — `list_tasks` noise.** Batch children now appear in `task list`. Acceptable, or should `list` filter them by
  default (with a `--all`)? (Recommend: show them; they carry `batch_id`/`item_id` for context.)

## Test strategy (TDD targets for the plan)
- **`is`/ids/store:** `BatchId` parse; `BatchRecord` round-trips through `SqliteTaskStore` + `MemoryTaskStore`;
  `TaskRecord` with `batch_id`/`item_id` round-trips; the additive migration is idempotent (re-open an old DB).
- **Admission loop:** with a fake `AgentRegistry`/recording sink — (a) a `concurrency=2` batch of 5 never exceeds 2
  in flight; (b) the **serve-wide** cap: two batches sharing `max_concurrent=3` never exceed 3 combined; (c)
  continue-others: a failing child doesn't abort the batch and the roll-up reports `failed=1`; (d) batch settles
  `Completed` when all terminal.
- **Cancel:** `cancel_batch` stops admission + cancels in-flight (assert the in-flight children's tokens fire) +
  settles `Cancelled`.
- **Crash-resume:** seed a `Working` batch with `items_json` of 4 items + 1 Completed child + 1 Working child (no
  checkpoint) + 2 un-created items; `resume_working_batches` re-admits the 2 tail items + re-runs the Working child,
  leaves the Completed one, all through the semaphore; batch settles `Completed`. Assert children are NOT double-
  spawned (item_id idempotency).
- **Cap-holds-on-boot:** seed two Working batches whose combined Working+tail children exceed `max_concurrent`;
  assert resume never exceeds the cap.
- **Inbound adapters:** `RunBatch` rejects an empty manifest / unknown workflow with the right JSON-RPC error;
  `BatchStatus` returns the roll-up; `CancelBatch` returns `true` for a Working batch.
- **CLI:** manifest parse (id default, input_file inline, dup-id reject, cwd-outside-root reject); `run-batch` prints
  a `BatchId`; the roll-up renders.

## Risks
- **Concurrency (the riskiest):** the admission loop + shared semaphore + per-batch cancel + crash-resume is the
  same class of bug-dense code as W3b. Mitigations: reuse the W3b `FuturesUnordered`-drain shape verbatim; the
  serve-wide cap is the single new global invariant — test it explicitly (two-batch + on-boot). The dual whole-
  branch review is the backstop (it caught the W3b/Slice-9 races the tests missed).
- **Migration:** additive only; follows the W3a/W3b idempotent pattern; test re-opening a pre-E3 DB.
- **Resume division (D6):** the subtle correctness point — batch children must skip the plain resume path or the cap
  breaks on boot. Called out as a dedicated test.

### SR-FIX-1 (BLOCKER — both lenses) — re-home the seam off the mcp-only Coordinator
`serve`'s request + boot paths run through `InboundServer` + **free functions** (`spawn_detached_workflow`,
`finalize_detached`; the "s8 T9 non-divergence" note @ `server.rs:2439` confirms Slice-8 is realized as shared
*functions*, not a shared *object*). Putting the cap on a `Coordinator` field strands it off the serve path.
**Resolution (Opus Option A — NOT the larger serve→Coordinator rewire, which both lenses say is its own slice):**
- The admission loop + resume = **free functions in `bridge-coordinator::batch` over a `BatchDeps` bundle**
  (mirroring `DetachedDeps` @ `detached.rs:257`).
- `batch_semaphore: Arc<tokio::sync::Semaphore>` is owned as a **field on `InboundServer`** (built in the serve
  wiring @ `main.rs:4421`, sized from `[batch]`), exactly as `workflow_cancels`/`progress_hubs` already live there;
  the `mcp` `Coordinator` owns its own (sized identically). `Coordinator::{run_batch,…}` become thin wrappers
  delegating to the same free functions over its semaphore. One semaphore **per process** = global within that
  process (serve OR mcp) — correct, since they are separate processes.
- The four `RunBatch/BatchStatus/BatchList/CancelBatch` inbound arms call the free functions via `srv`'s state
  (NOT a Coordinator), matching the existing `message/send` detached arm.

### SR-FIX-2 (BLOCKER — codex) — CAS status transitions (no Completed-over-Cancel clobber)
A plain `set_batch_status` races `Completed` (admission loop) vs `Canceling/Canceled` (`cancel_batch`). Mirror
`cancel_if_working`: store exposes **conditional** `cancel_batch_if_working(id, ts) -> bool` and
`settle_batch_if_status(id, expect, new, ts) -> bool` (both `WHERE status = ?`). The admission loop settles
`Completed` only `WHERE status='working'`; cancel transitions `working→canceling` then `canceling→canceled` only
from the expected prior state.

### SR-FIX-3 (MAJOR — codex) — boot-resume must cover `Canceling`, not only `Working`
A crash *after* `cancel_batch` wrote `Canceling` strands the batch (v1 resumed only `Working`). Boot scans
**`active_batches()` = `Working | Canceling`**: for `Working` → re-admit tail + re-run working children; for
`Canceling` → admit NO tail, cancel/settle still-working children, `settle_batch_if_status(canceling→canceled)`.

### SR-FIX-4 (MAJOR — both) — DB-enforced `(batch_id, item_id)` idempotency, not an app-level scan
App-level "scan children for this item_id before create" races a concurrent `resume_working_batches` admit
(two fresh TaskIds, same `(batch_id,item_id)` → double-spawn). The additive migration adds a **partial UNIQUE
index `(batch_id, item_id) WHERE batch_id IS NOT NULL`**; child-create for a batch item is idempotent on it
(insert-or-treat-conflict-as-existing within one tx).

### SR-FIX-5 (MAJOR — codex; RESOLVES Q1, supersedes v1's boot-only) — lazy CAS-settle in `batch_status`
Boot-only settle leaves a **live serve whose admission task panicked** reporting `Working` forever. `batch_status`
lazily CAS-settles `working→completed` (or `canceling→canceled`) when `pending=0 && running=0`, via
`settle_batch_if_status` (idempotent, race-free with the loop). The boot routine still settles too (covers a dead
serve).

### SR-FIX-6 (MAJOR — codex) — canceled-child accounting in the roll-up
`Canceled` is terminal but v1's `failed` counted only `Failed|Interrupted` → an out-of-band `cancel_task` on a
child would vanish from totals. `BatchSummary` buckets become explicit: `ok`=Completed, `failed`=Failed|Interrupted,
`canceled`=Canceled, `running`=Working, `pending`=no-row. **terminal = ok+failed+canceled** (the settle predicate).

### SR-FIX-7 (MAJOR — Opus) — compute `BatchSummary` once in `batch.rs`, keep the store at primitives
The `TaskStore` trait exposes only primitives — `get_batch`, `batch_children(&BatchId) -> Vec<TaskRecord>`,
`active_batches`, `list_batches`, `create_batch`, the two conditional setters. `BatchSummary` is a **pure function
over `(BatchRecord, Vec<TaskRecord>)` in `bridge-coordinator::batch`** (unit-testable; the roll-up math is not
re-implemented per store impl).

### SR-FIX-8 (MAJOR — codex; RESOLVES Q2) — admission ordering closes the visibility gap
Admit = **biased `tokio::select!` { cancel → abort, `batch_semaphore.acquire_owned()` → permit }** (cancel never
creates a row). After row-create + `workflow_cancels` token-register, **re-check batch/child state before spawn**;
if the batch is `Canceling` or the child was canceled in the gap → drop hub/token, release the permit, do not spawn.

### SR-FIX-9 (MAJOR→ruling — Opus & codex) — do NOT redefine `working_tasks()`
`working_tasks()` keeps its published contract (ALL working rows; 5 impls + SQLite @ sqlite.rs:590). The
"children skip the plain path" invariant is expressed by a **call-site partition** (or a new, explicitly-named
`working_non_batch_tasks()`), never by silently changing `working_tasks()`.

### SR-FIX-10 (MINOR — Opus) — one boot entry-point `resume_all`, routing by `batch_id`
Since SR-FIX-1/3 already rewire boot: a single free `resume_all(deps, cap)` (called from `main.rs:4466` serve AND
`Coordinator::resume()` mcp) (1) resumes `active_batches()` through the semaphore, (2) resumes working tasks
**partitioned** `batch_id.is_none()` via the existing path. "Every working task resumed exactly once, by exactly
one owner" is enforced in ONE place — this subsumes SR-FIX-9's partition + SR-FIX-3's active-scan + the
cap-holds-on-boot invariant.

### SR-FIX-11 (MINOR — Opus) — version-envelope `items_json`
Store `items_json = {"v":1,"items":[…]}` and version-gate on resume, identical to `encode_workflow_spec`'s
`{"v":N,…}` (`detached.rs:1401`, rejected-on-unknown-v @ `:1468`). Saves a migration when `BatchItem` grows a field.

### SR-FIX-12 (MINOR — Opus) — default-body the batch trait methods
The new `TaskStore` batch methods get **default bodies** (`Err(Unsupported)` / empty vec), following the
`journal_fold_inputs` default precedent (`task_store.rs:119`), so only `SqliteStore` + `MemoryTaskStore` implement
them for real; the 3 test-doubles (`FailingCheckpointStore` @ detached.rs:787, the workflow-producer + inbound
fakes) compile untouched.

### SR-FIX-13 (MINOR — codex; new D10) — `allowed_cwd_root` enforced SERVER-side, not by the CLI client
The thin `run-batch` client has no config/root (`run-workflow --serve` even rejects `--config`, `main.rs:780`). The
CLI validates manifest **shape** only (id-dedup, input XOR input_file, non-empty); the **server** re-validates each
item's `session_cwd` against `allowed_cwd_root` at submit and rejects the whole batch (`InvalidRequest`) before any
child. (v1's §"Manifest format" "rejected client-side" is corrected: shape client-side, cwd-root server-side.)

### SR-FIX-14 (NIT — codex) — signature + spelling corrections
`batch_status(&BatchId) -> BatchSummary` (v1 §"Coordinator API" wrongly said `-> BatchStatus`). `BatchStatus` enum
uses the codebase spelling: **`{Working, Completed, Canceling, Canceled, Failed}`** (American `Canceled`; `Canceling`
for in-progress) — supersedes v1's `Cancelled/Cancelling`.

### SR-FIX-15 (MINOR — Opus) — stop overselling "verbatim reuse"
The admission loop **awaits child completions to free permits** — a genuinely NEW completion-driven drain shape,
NOT a copy of `resume_working_tasks`' fire-and-forget (`drop(spawn_detached_workflow(...))` @ detached.rs:1651).
The §Risks "reuse the W3b FuturesUnordered-drain shape verbatim" is corrected to "follow the W3b drain *pattern*;
the await-completions admission loop is new code, tested directly."

### SR-FIX-16 (MINOR — Opus; new D9) — label the pool-not-bounded gap honestly
§Goal gains one sentence: **single (non-batch) runs are NOT counted against `max_concurrent`** — the cap bounds
concurrency *across batches*, not the whole agent pool. The eventual global migration (acquire a permit at the
`message/send` detached arm `server.rs:2495`) is **additive — not a one-way door**.

### SR-FIX-17 (NIT — Opus) — small hygiene
Group the bridge-private `RunBatch/…` arms with a comment distinguishing them from the `a2a::methods::*` A2A-spec
methods. Extract a free `validate_cwd(s: &str, root) -> Result<…>` (the inner check of `OpParams::validate_cwd`,
`params.rs:263`) so the batch path (per SR-FIX-13, server-side) and `OpParams` share one validator.

### Zero-value rejection (codex D2 note)
`max_concurrent`, `default_concurrency`, and a per-batch `--concurrency` must be `>= 1` — rejected at config-load
(the `[batch]` parse) and at submit. `default_concurrency` clamps to `<= max_concurrent`.

## v3 (BINDING — supersedes v2/v1 where they conflict)

Dual RE-REVIEW (codex xhigh + Opus). Both `needs-revision`, **strongly corroborating**. SR-FIX-1's re-homing onto
`InboundServer` is **RESOLVED by both** (`InboundServer` @ server.rs:75 already holds `task_store`,
`workflow_cancels`, `progress_hubs`, `workflows`, `executor`, `allowed_cwd_root` — the free-fn admission loop closes
over `srv` state with no `Coordinator`). The CORE design (D1–D8) is STABLE across both review rounds; v3 closes the
*second-order mechanics* that v2's fixes opened (the admission-loop cancel token, suppressed-row finalize, the claim
primitive, orphan/corrupt recovery). The fixes are concrete + bounded — both lenses say "v3 paragraph, not a spike."

### RR-FIX-1 (BLOCKER — codex) — the admission loop needs its OWN cancel token, not a child token
`workflow_cancels` is per-CHILD; a loop blocked on `batch_semaphore.acquire_owned()` cannot be woken by
`cancel_batch`. Add a per-batch `CancellationToken` (`batch_cancels: HashMap<BatchId, CancellationToken>`, housed in
`BatchRuntime` per RR-FIX-6). The admission/resume loop's biased `select!` arms on **its batch token**; `cancel_batch`
fires it (and CAS-flips state, RR-FIX-9). Registered before the loop spawns, removed on terminal.

### RR-FIX-2 (BLOCKER — codex; sharpens SR-FIX-8) — a suppressed-spawn child row must be durably finalized
SR-FIX-8's "re-check → drop hub/token, don't spawn" leaves the already-created child row `Working` **forever**:
`cancel_task` (server.rs:2715) relies on the spawned runner to write `Canceled`, and the executor writes terminal
**only if spawned** (detached.rs:1251). Fix: when the post-create re-check suppresses the spawn, **immediately
finalize the row durably via `finalize_detached(…, Canceled, hub)`** (so `terminal_seq` + hub cleanup stay correct) —
never just drop hub/token.

### RR-FIX-3 (BLOCKER — codex; completes SR-FIX-4) — a CLAIM primitive, not just the unique index
`UNIQUE(batch_id,item_id)` dedups ROWS, not RUNNERS; `create` only promises a duplicate-id error and `SqliteStore`
maps all insert failures to `StoreFailure` (sqlite.rs:382), so the admitter can't learn whether IT owns spawning. Add
a transactional store primitive **`claim_batch_child(batch_id, item_id, rec) -> ChildClaim { Created | ExistingWorking
| ExistingTerminal }`** (insert-or-select in one tx, SqliteStore + MemoryTaskStore). Admit/resume spawns **only** on
`Created`; `ExistingWorking` → already running (skip); `ExistingTerminal` → counted, not re-run. This is the real
idempotency keystone; the unique index is its backstop.

### RR-FIX-4 (MAJOR — both; completes SR-FIX-10) — `resume_all` must TOTAL-cover orphan working children
There is NO FK from `tasks.batch_id` to `batch.id` (existing child tables FK *into* `tasks`, sqlite.rs:125+). So a
`Working` child whose batch row is terminal/absent satisfies `batch_id.is_some()` (excluded from the plain partition)
AND its batch ∉ `active_batches()` (not re-driven) → **resumed by neither**. Fix: `resume_all`, after loading active
batch ids, **sweeps `Working` children whose batch ∉ active to `Interrupted`** (a warn), consistent with the existing
crash-sweep; the operator re-batches. Add a dedicated test.

### RR-FIX-5 (MAJOR — codex; completes SR-FIX-11) — corrupt/unknown-version plan recovery
`items_json` version-gating copies the detached envelope, but a batch has no `Interrupted` analog, so an
unparseable/unknown-`v` plan on an active batch strands it `Working/Canceling`. Fix: on resume parse-failure → cancel
any working children + **CAS the batch to `Failed`** with an `error` field; never strand.

### RR-FIX-6 (MAJOR — codex) — `Option<BatchRuntime>`, not a bare required semaphore
A bare `batch_semaphore: Arc<Semaphore>` field contradicts "`[batch]` absent → batch disabled" (a required semaphore
needs a size; 0 is rejected). `InboundServer`/`Coordinator` hold **`batch: Option<BatchRuntime>`** where
`BatchRuntime { semaphore: Arc<Semaphore>, default_concurrency: u32, max_concurrent: u32, batch_cancels:
Arc<Mutex<HashMap<BatchId, CancellationToken>>> }`. `run_batch`/`RunBatch` reject `None` →
`InvalidRequest{field:"batch (not configured)"}`. (Houses RR-FIX-1.)

### RR-FIX-7 (MAJOR — Opus; completes SR-FIX-1) — `BatchDeps` is a `DetachedDeps` SUPERSET, not a mirror
`DetachedDeps` (detached.rs:257) carries `{task_store, executor, workflows, workflow_cancels, progress_hubs, clock}`
and **lacks `allowed_cwd_root`**, which lives separately on `InboundServer` (`Option<String>`, server.rs:146) vs
`Coordinator` (`Option<SessionCwd>`, coordinator.rs:108 — note the type differs). `BatchDeps = DetachedDeps + {
batch: BatchRuntime, allowed_cwd_root: Option<SessionCwd> }`; BOTH wiring sites **normalize to `SessionCwd`** (serve
parses its `String` once, as `Coordinator::new` already does @ main.rs:4110).

### RR-FIX-8 (MAJOR — Opus; unifies SR-FIX-5/6) — one settle owner `is_settleable()`
Three sites compute "effectively complete" (lazy-settle in `batch_status`, the loop's drain, boot). Fix: the
`BatchSummary` pure-fn (SR-FIX-7) owns **`is_settleable(&BatchSummary) -> Option<BatchStatus>`** (and the bucket
math); ALL settle sites call it. The read-path lazy-settle is architecturally fine **because** the write is a guarded
CAS (`settle_batch_if_status`); the only residual risk was predicate drift — this removes it.

### RR-FIX-9 (MINOR — Opus; sharpens SR-FIX-2/3/8) — a SINGLE canceller of in-flight children
`cancel_batch` CAS-flips `working→canceling` and fires the **batch token** (RR-FIX-1) ONLY. The admission/resume loop
— observing `Canceling` on its next re-check, woken via its token — is the **SOLE canceller** of its in-flight
children (it owns their TaskIds in its `FuturesUnordered`). `cancel_batch` does NOT also enumerate-and-fire child
tokens (kills the snapshot-vs-create race). For a `Canceling` batch with no live loop (post-crash),
`resume_all`/`resume_working_batches` is the canceller.

### RR-FIX-10 (MINOR — both; corrects SR-FIX-12) — `StoreFailure` default + delegating-double rule
There is **no `BridgeError::Unsupported`** (error.rs:22); default-bodied batch methods return `Err(StoreFailure)`
(the `record_event_sequenced` precedent @ task_store.rs:201) or an empty vec for pure reads. Two of the three
test-doubles are **delegating wrappers** (`FailingCheckpointStore` forwards to an inner `MemoryTaskStore`,
workflow_producer.rs:2392) — so the rule is: **batch tests exercise batch ops through a REAL `SqliteStore`/
`MemoryTaskStore` directly, never a delegating double** (else the default body errors instead of delegating).

### RR-FIX-11 (MINOR — codex; completes Q7) — `task list` DTO carries `batch_id`/`item_id`
Today `list_tasks` exposes only id/workflow/state/updated (server.rs:3177). "Show children with context" requires
adding optional `batch_id`/`item_id` to `TaskStatusDto` + `list_tasks` + the CLI render.

### RR-FIX-12 (MINOR — Opus) — size both `BatchRuntime`s from ONE `[batch]` helper
serve + mcp must build `BatchRuntime` from the same `[batch]` parse (mirror `resume_cap`, computed identically @
main.rs:4074/:4352) so the two per-process sizings cannot diverge.

### RR-FIX-13 (NIT — both) — `validate_cwd(s, root, field_label)`
The extracted free helper takes the caller-facing field label so it preserves `a2a-bridge.cwd` (A2A, server.rs:3339)
vs `cwd` (`OpParams`, params.rs:270) error fields. Batch passes its own label.

### RR-FIX-14 (NIT — Opus) — name the offending item on manifest reject
`InvalidRequest{field:&'static str}` (error.rs:27) cannot name which item failed. Use `ConfigInvalid{reason:String}`
(error.rs:70) for the whole-batch manifest/cwd reject so the operator gets the item id.
