## Review: rs-05 — `Manager::get_or_spawn`

**BLOCKER (WRONG) — TOCTOU double-spawn; the lock is the wrong seam.** `src/lib.rs:28-33`. The critical section is split across three independent lock acquisitions with the lock *released* across the expensive `spawn`. The guard from `self.sessions.lock().unwrap().contains_key(id)` (line 29) is a temporary dropped at the end of the `if` condition, so `spawn` (line 30) and `insert` (line 31) run outside any single held guard. Concrete failure: two handlers call `get_or_spawn("x")` concurrently, both observe the key absent, both call `spawn` → `spawned` reaches 2, two real agent processes start, and the second `insert` overwrites (and silently drops) the first `Session`, orphaning its already-started process. This violates *both* stated invariants ("at most ONE session ever spawned per id" and "concurrent calls collapse to a single spawn"). The root architectural gap: the map has only two states — *present* or *absent* — with no *in-flight/pending* state, so the design literally cannot express "another thread is already spawning this id; wait for it." **Fix direction:** hold one guard across check+insert (`entry(...).or_insert_with(...)`) *or*, to avoid serializing all spawns behind the map lock, insert a pending marker (`Arc<OnceLock<Session>>` / a shared future) under the lock and let only the winner spawn outside it. Note the tension: the naive single-guard fix holds the lock across the slow `spawn`, blocking every other id's lookup for the duration — which is why the missing "pending" seam, not just a wider lock, is the real fix.

**MAJOR (WRONG) — `get_or_spawn` returns `()`, and the storage type forecloses ever returning the session.** `src/lib.rs:28`. The doc says "Return the session for `id`," but the signature returns nothing, so a caller must do a *second* `lock()`-and-fetch — reintroducing a race window and duplicating lookup logic. Worse, it *cannot* be fixed by just changing the return type: `Session` is stored by value behind a `Mutex`, so no `&Session` can outlive the guard. The abstraction doesn't fit its own "get" contract. **Fix direction:** store `HashMap<String, Arc<Session>>` and return a cloned `Arc<Session>`, so sessions are shareable and the winner's session is authoritative for all callers.

**MAJOR (SMELL) — the "spawn-once" invariant is split across two independent mutexes.** `src/lib.rs:9-10, 23, 31`. `spawned` (the counter that *is* the invariant being asserted) and `sessions` (the actual cache) are protected by different locks and updated in different statements, so they can drift — under the race above, `spawned == 2` while the map holds one entry. No single lock owns "spawned exactly once per id." **Fix direction:** derive the count from the map, or increment it under the same guard as the `insert`, so one lock protects the whole invariant.

**MINOR (SMELL) — panic-on-poison in a long-lived, widely-shared service.** `src/lib.rs:29, 31, 36`. `.lock().unwrap()` means one panic while holding either lock poisons it, and every subsequent `get_or_spawn` from every request handler then panics — a single-session failure escalates to a service-wide outage. **Fix direction:** recover from poison (`.lock().unwrap_or_else(|e| e.into_inner())`) or move to a lock strategy that doesn't couple one handler's panic to all others.

---

**Verdict:** BLOCKER — the check-then-spawn race double-spawns agents and violates both contract invariants; the underlying architectural miss is a "pending/in-flight" seam, and the by-value storage plus `()` return signature mean the abstraction can't even hand back the session it promises.
