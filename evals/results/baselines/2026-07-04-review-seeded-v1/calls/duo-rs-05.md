The correctness lens and the architecture lens agree fully — the core race is the same finding from both. Here's the merged review.

**BLOCKER — WRONG — TOCTOU double-spawn race, `src/lib.rs:28–31` (`get_or_spawn`).** The cache check and the insert run under two *separate* `sessions` locks. The `contains_key` guard's lock guard is a temporary that drops at the end of the `if` condition, so the mutex is released before `spawn` and re-acquired for `insert`. Two concurrent `get_or_spawn("a")` calls both observe the key absent, both call `spawn("a")` (two real agent processes), and each inserts — the second overwrites the first, orphaning a live process while `spawn_count()` reads 2. This directly violates the primary contract ("at most ONE session per id; concurrent same-id calls collapse to one spawn"). The architectural root is that a map-wide `Mutex<HashMap>` can only guard map operations, not the spawn *lifecycle* — there is no per-id single-flight seam. **Fix:** reserve the slot atomically under one lock — e.g. `HashMap<String, Arc<OnceCell<Session>>>` (or an entry placeholder) so the slot is claimed under the lock and the expensive `spawn` resolves the cell *outside* the lock, exactly once. Do NOT "fix" by simply holding the lock across `spawn` — that serializes every id's spawn and blocks all readers for the duration of a real process launch, trading a race for head-of-line blocking.

**MAJOR — WRONG — `get_or_spawn` returns `()`, `src/lib.rs:28`.** The doc says "Return the session for `id`" but the signature yields nothing, so the "get" half of get-or-spawn does not exist. A caller needing the authoritative cached session must re-lock and look it up separately — duplicating the critical section and re-opening a race (nothing guarantees the entry still exists between the call and the caller's own lookup). Compounding this, the map owns a bare `Session` (not `Arc`/`Clone`), so there is no way to hand a cached session out by value at all — the ownership model cannot satisfy the "many handlers share one session" requirement it was built for. **Fix:** store and return `Arc<Session>` (`get_or_spawn -> Arc<Session>`).

**MAJOR — SMELL — no error seam on a fallible spawn, `src/lib.rs:22 & 28`.** The doc says `spawn` "starts a real agent process" — inherently fallible (exec failure, resource exhaustion) — yet `spawn -> Session` and `get_or_spawn -> ()` give a spawn failure nowhere to go but `panic!`, unwinding a request handler with no typed failure for a recovery caller to branch on. **Fix:** `spawn -> Result<Session>`, `get_or_spawn -> Result<Arc<Session>>`.

**MAJOR — SMELL — missing concurrency regression test, no test files in fixture.** There is no thread/barrier test asserting that many concurrent `get_or_spawn("x")` calls yield exactly one spawn — the core acceptance behavior, and a test that would fail against the current code. **Fix:** add a barrier-synchronized multi-thread test that calls `get_or_spawn` for the same id from N threads and asserts `spawn_count() == 1`.

**MINOR — SMELL — `.unwrap()` on every lock propagates poisoning into a permanent wedge, `src/lib.rs:23,29,31,36`.** Any panic while a guard is held poisons the mutex, turning every later `get_or_spawn`/`spawn_count` on the *shared* Manager into a panic. The window is small today (spawn runs unlocked) but the single-flight fix above widens it by holding a slot under lock. **Fix:** a poison-recovery policy (`unwrap_or_else(|e| e.into_inner())`) or a non-poisoning lock (`parking_lot`).

**MINOR — SMELL — split invariant between `spawned` and `sessions`, `src/lib.rs:10 & 23`.** The counter and the session map are guarded by separate mutexes with no atomic tie, so `spawn_count()` cannot be reasoned about relative to `sessions.len()` — under the current race they already diverge (count 2, one entry). Harmless as telemetry now, a correctness trap the moment any code asserts `spawn_count == sessions.len()`. **Fix:** derive the count from the map, or bump it inside the same critical section that claims the slot.

---

**Verdict:** Do not merge — fix the BLOCKER double-spawn race (with the `Arc<OnceCell>` single-flight seam) and the two MAJOR contract gaps (`Arc<Session>` return, `Result` error seam); add the concurrency regression test to lock it in.
