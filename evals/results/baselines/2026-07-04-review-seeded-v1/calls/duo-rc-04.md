**BLOCKER — `collect()` gates completion on message *count*, not node *identity* → silent result corruption on any duplicate send.** `src/lib.rs`, `collect()` (`self.rx.into_iter().take(self.nodes)`, ~line 35–37). `take(nodes)` caps on how many messages arrived, never on which distinct nodes reported. Concrete: with nodes A, B, C (`nodes = 3`), if A double-sends (retry/re-drive/caller bug) and C never lands, `collect()` reads `[A, A, B]`, stops, and returns a full-length `Vec` — a duplicate plus a silently missing node. Callers can't even detect it via `len() == nodes`, because the length is right. The `NodeResult.node` field exists precisely to key results per node but is never used. *Fix:* collect into a map keyed by `node`, track the expected node-id set, and return an error on any duplicate or missing node. (Codex and Claude both flag this; Claude's BLOCKER rating is correct over Codex's MAJOR — silent wrong output with no caller-side signal is the most severe defect here.)

**BLOCKER — liveness depends on an invariant the Collector can't enforce; one un-sent, un-dropped sender hangs `collect()` forever.** `src/lib.rs`, same drain line. `into_iter()` blocks on `recv()` and only terminates when `nodes` messages arrive *or* all senders drop. If a node fails before sending **and** its cloned `Sender` is still held somewhere (stuck task, retry/join queue, leaked spare clone), the count never reaches `nodes` and the channel never disconnects → permanent hang, no timeout, no cancellation seam. The doc's "each node sends exactly one result" is a liveness *precondition* nothing in this unit guards. *Fix:* drive completion off the known node-id set with a bounded/timeout recv so a missing node becomes a returned error, not an indefinite block.

**MAJOR — `collect() -> Vec<NodeResult>` cannot signal incompleteness.** `src/lib.rs`, `collect()` return (~line 34–37). In the benign path (a node's sender drops without sending), the channel disconnects early and `collect()` returns a short `Vec` with no error — the type erases "all nodes reported" vs "some silently vanished," and per the first BLOCKER even a length check is unreliable. *Fix:* return `Result<Vec<NodeResult>, Incomplete { missing, duplicated }>` so partial completion is a typed outcome.

**MAJOR (conditional on async runtime) — blocking `std::sync::mpsc::recv` is the wrong primitive for a workflow-DAG executor.** `src/lib.rs:1` and the drain line. `into_iter()`'s blocking `recv()` parks the calling thread; on a tokio worker it holds a runtime thread for the whole gather and can starve or deadlock the executor whose nodes are the very producers it awaits. The "workflow node" framing strongly implies async, but the self-contained fixture doesn't confirm it — hence conditional. *Fix:* use `tokio::sync::mpsc` with async recv, or confine the blocking drain to `spawn_blocking`.
