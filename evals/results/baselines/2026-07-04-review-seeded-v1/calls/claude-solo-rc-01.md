First, the correctness baseline (so the architecture findings are properly framed): the `select!` logic is sound for its contract. `biased;` makes the abort-first guarantee load-bearing and correct — when a `Some(())` and a `Some(d)` are simultaneously ready, cancel wins and the delta is not drained. A correctness-only pass over the diff should PASS it. Here is what that pass misses.

---

**MAJOR — No failure state in the type-space; channel-close conflates "agent finished" with "agent died."** `src/lib.rs:4-7,32-33`. `TurnOutcome` is `Completed | Cancelled`, and `deltas: Receiver<String>` cannot carry an error. The only "done" signal is `deltas.recv() == None`, which the code maps unconditionally to `Completed(acc)`. In the real bridge a mid-turn agent crash drops the sender → `recv() == None` → a **truncated partial output is reported as a fully successful turn** (silent truncation). This is the classic architectural gap a correctness pass over the contract won't catch, because the contract itself defines None-as-completion. The first future change that needs to surface an agent error (this project already has `AgentCrashed{reason}` inbound) has nowhere to put it. Fix direction: `deltas: Receiver<Result<String, TurnError>>` and/or add `TurnOutcome::Failed(..)`, so completion and failure are distinct returns rather than one collapsed `None` arm.

**MINOR — `Cancelled` discards the accumulated transcript.** `src/lib.rs:6,27`. On cancel, `acc` is dropped and the variant carries no payload. In a system whose whole ethos is durable partial state (journals/checkpoints/resume), a cancel that throws away what was already streamed forecloses persist-on-cancel. Adding it later is a breaking change to a `pub enum`. Fix direction: `Cancelled(String)` (or return `(TurnOutcome, String)`) now, while the API has no external callers to break.

**MINOR — Whole-turn buffering with no bound is baked into the return type.** `src/lib.rs:5,17,31`. `Completed(String)` forces the entire turn to be accumulated in RAM, and `acc.push_str` has no size cap or backpressure. A runaway/chatty agent (the very threat the `biased` comment cites for cancellation) drives unbounded memory growth → OOM, and cancellation only helps if a cancel is actually sent. Fix direction: cap accumulated size (return/err past a threshold) or stream deltas to a sink incrementally rather than materializing the full turn in the outcome.

**MINOR — Receivers taken by value make this single-use per turn, which sits awkwardly against "warm."** `src/lib.rs:16`. `run_turn` consumes both `Receiver`s and drops them on return, so the delta channel cannot survive into the next turn. That's fine if every turn mints a fresh channel, but there is no seam expressing which lifetime is intended, so a future "reuse the session's delta channel across turns" change silently can't be built on this signature. Fix direction: document/encode the ownership decision (borrow if per-session, keep by-value if genuinely per-turn) so the warm-vs-turn boundary is explicit.

---

**Verdict:** No correctness BLOCKER — the diff conforms to its contract — but the outcome/channel type-space is too narrow: it cannot represent agent failure (MAJOR, silent truncation) and forecloses partial-output recovery and bounded buffering; widen the types before external callers appear.
