I'll examine the artifact and its surrounding structure before writing the review.## Review: rs-09 — `frame::encode`

**BLOCKER (WRONG) — `payload.len() as u16` silently truncates any payload ≥ 64 KiB** (`src/lib.rs:20`). The contract permits payloads up to 1 MiB, but `u16` maxes at 65 535. `as u16` is a wrapping truncation (never panics, even in debug), so a 100 000-byte payload yields `len = 100_000 % 65_536 = 34_464`. `to_bytes` then writes a header claiming 34 464 bytes in front of a 100 000-byte body. The reader consumes 34 464 bytes as the payload, then interprets byte 34 465 (mid-payload) as the next frame's length header — the stream desynchronizes and **every subsequent frame on the connection is corrupted**, with no error surfaced. Trigger: any single MCP message the contract explicitly allows (a 200 KB tool result, a large `initialize` payload). Fix: the length must be carried in a field wide enough for the domain, and the cap enforced — see next.

**BLOCKER (WRONG, root cause) — the wire format itself cannot represent the stated payload range** (`src/lib.rs:3`, the 2-byte header; `src/lib.rs:4`, `pub len: u16`). A 2-byte big-endian header addresses at most 65 535 bytes; the contract says 1 MiB (1 048 576, needs 21 bits). The type `u16` *encodes an invariant* — "payload ≤ 64 KiB−1" — that flatly contradicts the domain. `encode` cannot be made correct by patching the cast alone; the field width is the defect and the `as u16` is only where it surfaces. Fix direction: widen the header to 4 bytes (`len: u32`) and update `to_bytes` accordingly; then bound-check against the 1 MiB cap on the way in.

**MAJOR (design) — no seam owns the `header == payload.len()` invariant** (`src/lib.rs:4-5`, `13-14`). `Frame` exposes `pub len` and `pub payload`, so the length and body can drift independently; `to_bytes` trusts `self.len` blindly instead of deriving it. `encode` is meant to be the constructor that establishes the invariant, but because the fields are public it has no exclusive authority — any caller can build a mismatched frame, and a future edit that sets `len` by hand reintroduces the desync bug this task is about. Fix: don't store `len` at all — make `payload` private and have `to_bytes` compute `payload.len()` at serialization time (with a checked conversion), collapsing the invariant to a single derivation site.

**MAJOR (design) — infallible signature hides a real failure mode** (`src/lib.rs:19`). `encode(payload) -> Frame` promises the operation always succeeds, but "payload too large for the frame format" is a genuine, reachable failure that the type currently launders into silent corruption. An infallible API gives callers no place to reject or split an oversized message. Fix: `encode(payload) -> Result<Frame, FrameError>` (or a checked builder), returning an error when `payload.len()` exceeds the header's representable max — so the failure is handled at the boundary, not on the wire.

**MINOR (smell) — denormalized state (`len` stored alongside `payload`)** (`src/lib.rs:4`). Keeping a cached length next to the source of truth invites the two to diverge on every future mutation path. Derive length on demand rather than caching it; this also removes the class of bug entirely rather than fixing one instance.

**Verdict:** Reject — the `as u16` cast corrupts the transport stream for any payload ≥ 64 KiB, and the underlying 2-byte header / `u16` field cannot represent the contract's 1 MiB range, so the fix requires a wire-format change plus a fallible, invariant-owning constructor, not just a cast tweak.
